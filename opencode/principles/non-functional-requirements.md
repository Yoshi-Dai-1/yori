# 非機能要件（Non-Functional Requirements）

## このファイルの目的

「何ができるか」は機能要件で定義する。
「どれくらいの品質で動くべきか」を定義するのが非機能要件。

機能要件の例：「株式を検索できる」
非機能要件の例：「株式検索は500ms以内に結果を返す。同時接続100ユーザーでも性能を維持する」

非機能要件は**数値で定義しなければ意味がない**。
「高速に」「安全に」「安定して」は要件ではなく希望。

---

## なぜ非機能要件を事前に定義するのか

後回しにすると必ず以下が起きる：

- 本番リリース直前に「重い」と発覚し、アーキテクチャを作り直す
- セキュリティ要件が後から追加され、設計全体に影響する
- 「どのくらい速ければよいか」の合意がなく、判断基準がない

AIにコード生成を依頼するときも同じ。
非機能要件が未定義だと、AIは「動けばよいコード」を生成する。
「1000件のデータを50ms以内に処理するコード」と指示すれば、AIは適切なアルゴリズムを選択できる。

---

## フェーズと規模による定義の優先度

「すべてを今決める必要はない」がこのセクションの要旨。
後回しにした項目は、**その理由をARCHITECTURE.mdに1行書く**（意図的な妥協として記録する）。
書かずに放置すると、将来「なぜ未定義なのか」が分からなくなる。

| カテゴリ | プロトタイプ・個人開発 | 本番リリース前 | 商用・チーム開発 | 備考 |
|---------|----------------------|--------------|----------------|------|
| パフォーマンス | スキップ可 | 主要APIのp95だけ定義 | SLI/SLOとして定義 | プロトタイプ完了後に実測して決める |
| スケーラビリティ | ページネーションのみ決める | ユーザー数・データ量を定義 | エラーバジェット管理 | 過剰設計の最多発生源 |
| 可用性 | スキップ可 | 目標可用性とバックアップだけ | SLA定義・インシデント管理 | 商用は99.9%以上が標準 |
| セキュリティ | **APIキー管理とHTTPSは必須** | 全項目を必ず定義 | 定期監査・ペネトレーションテスト | 後から足すと設計全体に波及する |
| 保守性（ログ） | console.logで可 | ログレベルと出力先を定義 | 監査ログ・保存期間の法的対応 | 本番障害の調査に直結する |
| コスト | 無料枠のサービスを選ぶ | 請求アラートを設定 | 予算管理・SLA違反時の補償 | 個人開発で最も見落とされる |

| 法的・コンプライアンス | OSSライセンス確認のみ | プライバシーポリシー・業界規制を確認 | 全法令への準拠・監査ログ保存 | 後から発覚すると対応コストが指数的に増大する |
| アクセシビリティ | 対象外（CLIやAPI）またはLevel A最低限 | WebFEはLevel A必須 | Level AA目標・法的義務確認 | 後付けはUI全体の改修になる |
| コード品質 | ファイル行数のみ意識 | リンター・フォーマッターをCIに組み込む | カバレッジ目標・複雑度管理 | 定義しないとAIが「動けばよい」基準で生成する |
| 国際化（i18n） | UTF-8統一のみ | 多言語展開する場合に定義 | 全項目を定義 | 後から対応するとUI全体に影響する |
| テスト戦略 | ビジネスロジックのユニットテストのみ | テストフレームワークと方針を定義 | テストピラミッド全層を定義 | 方針なしだとテストが散らかる |

商用・チーム開発では、上記に加えて**開発プロセス要件**（ブランチ戦略・変更管理・インシデント管理・監査ログ）の定義が必要になる。
これらは非機能要件ではなく運用ガバナンスの領域であり、`.opencode/standards/principles/commercial-operations.md` で定義する。

**セキュリティだけは例外**：フェーズに関わらず最低限の定義が必要。
「APIキーを環境変数で管理する」「通信はHTTPS」は初日から守る。
この2点だけはプロトタイプ段階でもスキップしない。

## 非機能要件の11カテゴリ

### 1. パフォーマンス（Performance）

**定義すべき指標：**

| 指標 | 意味 | 例 |
|------|------|-----|
| レスポンスタイム | 1リクエストの応答時間 | API応答：p95で500ms以内 |
| スループット | 単位時間あたりの処理件数 | 100 req/sec を処理できる |
| レイテンシ | 処理開始までの遅延 | DB読み取り：10ms以内 |

**pXX表記について：**
「p95で500ms以内」＝全リクエストの95%が500ms以内に返る（残り5%は超えてよい）。
「平均値」ではなくパーセンタイルで定義するのが実用的。外れ値を隠さないため。

**テンプレート：**
```
[ ] 主要API（一覧取得）：p95で ___ms 以内
[ ] 主要API（検索）：p95で ___ms 以内
[ ] ページ初期表示：___ 秒以内（Core Web Vitals: LCP）
[ ] バッチ処理（___件）：___ 分以内
```

---

### 2. スケーラビリティ（Scalability）

**定義すべき指標：**

同時接続ユーザー数・データ量の増加に対して、システムがどう振る舞うべきか。

```
現在の想定規模：
  同時接続ユーザー：___人
  データ件数（初期）：___件
  データ件数（1年後想定）：___件

スケーリング方針：
  [ ] 垂直スケーリング（サーバーのスペックを上げる）
  [ ] 水平スケーリング（サーバーを複数台に増やす）
  [ ] スケーリング不要（個人利用・社内限定）

データ増加への対応：
  [ ] インデックス設計（どのカラムにインデックスを貼るか）
  [ ] ページネーション（一度に返すデータ件数の上限：___件）
  [ ] キャッシュ戦略（何をキャッシュするか・TTL：___秒）
```

**個人開発での現実的な考え方：**
スケーラビリティを過剰設計するのは時間の無駄。
しかし「ページネーションなしで全件取得」「インデックスなし」は、データが増えた時点で致命的になる。
最低限の設計（ページネーション・主要カラムへのインデックス）は最初から入れる。

### 3. 可用性・信頼性（Availability / Reliability）

**定義すべき指標：**

| 指標 | 意味 | 例 |
|------|------|-----|
| 可用性（Availability） | システムが稼働している時間の割合 | 99%（年間87時間のダウンタイム許容） |
| RTO（Recovery Time Objective） | 障害発生から復旧までの目標時間 | 4時間以内に復旧 |
| RPO（Recovery Point Objective） | 障害発生時に許容できるデータ損失の範囲 | 直近24時間分のデータは保護する |

**可用性の目安：**
```
99%    → 年間87時間のダウンタイム許容（個人開発・社内ツール）
99.9%  → 年間8.7時間（小規模商用サービス）
99.99% → 年間52分（重要な商用サービス）
```

**テンプレート：**
```
目標可用性：___%
RTO（復旧時間目標）：___ 時間以内
RPO（データ損失許容範囲）：___ 時間分

障害対応方針：
  [ ] バックアップ頻度：___ 時間ごと
  [ ] バックアップ保存先：___
  [ ] 障害検知方法：___（メール通知 / Slack通知 / etc.）
```

### 4. セキュリティ（Security）

セキュリティは「後で追加する」が最も危険な考え方。
設計の最初から組み込む。

**このセクションを記入する前に：**
`.opencode/standards/principles/security-requirements.md` の判断ステップを実行し、
このプロジェクトの対応レベル（Lv.1〜4）を導出してから記入する。
対応レベルが導出できない場合（project-definition.md が不足している等）は
`@security-designer` を呼び出す。

**定義すべき項目：**

**認証・認可：**
```
認証方式：
  [ ] セッションベース（Cookie）
  [ ] トークンベース（JWT）
  [ ] OAuth2.0（外部サービス連携）
  [ ] 認証なし（公開API）

認可（誰が何をできるか）：
  ロール定義：
    - ___ ロール：___ の操作が可能
    - ___ ロール：___ の操作が可能
```

**データ保護：**
```
機密データの定義（何が機密情報か）：
  [ ] パスワード → bcrypt/argon2でハッシュ化（平文保存禁止）
  [ ] APIキー → 環境変数で管理（コードに直書き禁止）
  [ ] 個人情報 → ___（暗号化 / マスキング / etc.）

通信の保護：
  [ ] HTTPS必須（HTTP禁止）
  [ ] APIキーをURLに含めない（ヘッダーで送る）
```

**入力値の検証：**
```
すべての外部入力を信頼しない原則（Trust No Input）：
  [ ] フロントエンドのバリデーションはUXのため（セキュリティのためではない）
  [ ] バックエンドで必ずバリデーションする
  [ ] SQLインジェクション対策：ORMのパラメータバインディングを使う
  [ ] XSS対策：ユーザー入力をそのままHTMLに出力しない
```

**依存ライブラリの管理：**
```
[ ] npm audit / pip audit を定期実行（または CIに組み込む）
[ ] 脆弱性が発見されたライブラリは速やかに更新する
[ ] 使っていないライブラリは削除する
```

### 5. 保守性（Maintainability）

コードの保守性は `principles/` の内容でカバー済み。
ここではシステムレベルの保守性を定義する。

**可観測性（Observability）について：**
可観測性とは「本番で何が起きているかを把握できる状態」。
「動いているように見える」と「健全に動いている」は別物。
実装の詳細は `.opencode/standards/principles/resilience.md`（Layer2: Detection）に集約している。

可観測性の3本柱：
```
ログ（Logs）       → 何が起きたか（イベントの記録）
メトリクス（Metrics）→ どのくらいの規模で起きているか（数値の時系列）
トレース（Traces） → どこを通って起きたか（処理の経路）
```

最低限の実装（個人・小規模）：
```
ログ     → 構造化ログ（JSON形式）・適切なログレベル・機密情報を含まない
メトリクス → クラウドの標準メトリクス（CPU・メモリ・レイテンシ）
トレース  → エラー監視ツール（Sentry等）でエラーのスタックトレースを取得
```

**ログ設計：**
```
ログレベルの定義：
  ERROR  → 即時対応が必要な障害（例外・APIエラー・DB接続失敗）
  WARN   → 注意が必要だが継続可能な状態（リトライ成功・閾値超過）
  INFO   → 通常の操作記録（リクエスト受信・処理完了）
  DEBUG  → 開発時のデバッグ情報（本番では出力しない）

必ずログに含める情報：
  [ ] タイムスタンプ（ISO 8601形式）
  [ ] ログレベル
  [ ] 処理内容（何が起きたか）
  [ ] コンテキスト（どのユーザー・リクエストIDなど）
  [ ] エラーの場合はスタックトレース

禁止事項：
  [ ] パスワード・APIキーをログに出力しない
  [ ] 個人情報をログに出力しない（必要な場合はマスキング）
```

**エラーハンドリングの方針：**
```
[ ] エラーはユーザーに分かりやすいメッセージで返す
[ ] エラーの詳細（スタックトレース）はユーザーに見せない
[ ] すべてのエラーはログに記録する
[ ] 想定外のエラーはアラートで通知する
```

### 6. コスト（Cost）

個人開発でも無視できない。無料枠を超えた時点で予想外の請求が来ることがある。

```
インフラコストの上限（月額）：___円

コスト管理方針：
  [ ] クラウドの請求アラートを設定する（上限の80%で通知）
  [ ] 使っていないリソースを定期的に削除する
  [ ] 開発環境と本番環境のスペックを分ける

外部APIのコスト：
  APIサービス名 | 無料枠 | 想定使用量 | 超過時の料金
  ___________  | _____ | ________ | __________
```

### 7. 法的・コンプライアンス（Legal / Compliance）

セキュリティと異なり、これは「技術で解決できない」カテゴリ。
発覚のタイミングが遅いほど対応コストが指数的に増大する。
「自分のプロジェクトには関係ない」と思い込むことが最大のリスク。

```
個人情報・データの取り扱い：
  [ ] 個人情報（氏名・住所・メールアドレス等）を収集するか
  [ ] 個人情報保護法（日本）/ GDPR（EU）の適用対象か
  [ ] プライバシーポリシーの作成・公開が必要か
  [ ] データの保存期間・削除ポリシーを定義しているか

業界固有の規制：
  [ ] 金融情報を扱う → 金融商品取引法・投資助言業の登録要否を確認
  [ ] 医療・介護情報を扱う → 要配慮個人情報として厳格な管理が必要
  [ ] その他業界固有の規制：___

外部サービス・ライブラリのライセンス：
  [ ] 使用するOSSのライセンスを確認しているか（GPL汚染に注意）
  [ ] 商用利用する外部APIの利用規約を確認しているか
  [ ] データソースの再配布・二次利用の可否を確認しているか

判断方針：
  技術的な判断はAIが支援できるが、法的判断はAIに委ねない。
  グレーゾーンがある場合は専門家（弁護士・行政書士）に確認する。
  「おそらく大丈夫」で進めた場合は必ず decisions/ に理由を記録する。
```

### 8. アクセシビリティ（Accessibility）

Webフロントエンドを含むプロジェクトでは必須の非機能要件。
アクセシビリティを後付けで対応するコストは初期対応の10倍以上になる。
デスクトップアプリ・モバイルアプリにも各プラットフォームのアクセシビリティ要件がある。

**適用対象の判断：**

```
Webフロントエンドあり → WCAG 2.1 を基準に適合レベルを決める
モバイルアプリ（iOS）   → Apple Human Interface Guidelines のアクセシビリティ要件
モバイルアプリ（Android）→ Android Accessibility guidelines
デスクトップアプリ      → プラットフォーム固有のアクセシビリティ要件
CLIツール              → 通常は対象外（スクリーンリーダーとの互換性は検討する）
APIのみ                → 対象外
```

**WCAGの適合レベル：**

| レベル | 内容 | 採用する場面 |
|--------|------|------------|
| Level A | 最低限の基準（26項目） | すべてのWebプロジェクトで最低限満たす |
| Level AA | 実質的な標準（13項目追加） | 公開Webサービスはこのレベルを目標にする |
| Level AAA | 最高水準（28項目追加） | 公共機関・アクセシビリティが特に重要な場合 |

政府・公共機関向けプロジェクトでは、法的にLevel AA準拠が求められる場合がある。
最新の要件を確認する：`WCAG 2.2 requirements [現在年]` / `JIS X 8341 [現在年]`

**実装チェックリスト（Webフロントエンド）：**

```
Level A の主要項目：
  [ ] 画像に代替テキスト（alt属性）がある
  [ ] フォームのラベルが入力欄と紐付いている（label要素・aria-label）
  [ ] キーボードだけで全機能が操作できる
  [ ] 自動再生する音声・動画には停止手段がある

Level AA の主要項目：
  [ ] テキストのコントラスト比が 4.5:1 以上（大きなテキストは 3:1）
  [ ] フォーカスが視覚的に分かる
  [ ] エラーメッセージが具体的で修正方法を示している
  [ ] ページの言語が指定されている（lang属性）
```

法的義務がある業種（公共機関・障害者向けサービス等）は専門家に確認する。

### 9. 国際化・ローカライゼーション（i18n / l10n）

後から対応するとコードベース全体に影響するため、設計初期に方針を決める。

**適用対象の判断：**
```
日本国内のみ・日本語のみ → 最低限：文字コードをUTF-8に統一。タイムゾーンをAsia/Tokyoで統一。
複数言語・複数地域展開  → 以下のチェックリスト全項目を定義する。
```

**チェックリスト（複数言語・地域展開の場合）：**
```
文字・エンコーディング：
  [ ] 全文字列はUTF-8で統一する
  [ ] DBのカラム文字セットはutf8mb4（絵文字対応）を使用する

テキスト管理：
  [ ] UIテキストはソースコードにハードコードしない
  [ ] 翻訳ファイルの管理方法を決める（フォーマット・ファイル構造）
  [ ] 翻訳の自動化方法を決める（Google翻訳API / 専門翻訳者 / コミュニティ）

日時・数値・通貨：
  [ ] タイムゾーンはUTCで保存し、表示時にユーザーのローカルタイムに変換する
  [ ] 日付フォーマットはロケールに応じて表示を変える（YYYY/MM/DD vs MM/DD/YYYY）
  [ ] 通貨・数値の桁区切り・小数点はロケールに応じて表示を変える

レイアウト：
  [ ] RTL（右から左に読む言語：アラビア語・ヘブライ語等）の対応要否を決める
  [ ] テキスト長の変化によるレイアウト崩れを考慮する（日本語→英語で文字数が増える場合がある）
```

最新の実装方法はフレームワークの公式ドキュメントで確認する：
`[フレームワーク名] i18n internationalization [現在年]`

### 10. テスト戦略（Testing Strategy）

「テストカバレッジ数値」はコード品質カテゴリで定義するが、「何をどのレイヤーでテストするか」は戦略として設計初期に決める。
決めないとAIが実装するたびにテストの粒度・方針がばらつく。

**テストピラミッドの方針：**
```
E2Eテスト（少数）
  - クリティカルなユーザーフロー（購入・ログイン・コア機能）のみ対象
  - 実行コストが高いため最小限に絞る
  - ツール：[Playwright / Cypress / Detox（モバイル）等]

インテグレーションテスト（中程度）
  - APIエンドポイント・DBアクセス・外部サービス連携を対象
  - モック方針：外部サービスはモック、DB は実際の接続を使う（テストDB）

ユニットテスト（多数）
  - ビジネスロジック・純粋関数・ユーティリティを対象
  - UIコンポーネントのユニットテストは「ロジックを含む場合のみ」作成する
```

**テスト方針の決定項目：**
```
  [ ] テストフレームワーク：[Jest / Vitest / pytest / Go test 等]
  [ ] モックライブラリ：[Vitest mock / unittest.mock 等]
  [ ] テストDB：[インメモリ / Dockerコンテナ / テスト専用スキーマ]
  [ ] CIでのテスト実行：[プルリクエスト時 / マージ時 / 定期実行]
  [ ] テストが失敗した場合のマージポリシー：[ブロック / 警告のみ]
```

テストフレームワークの最新状況はフレームワーク公式ドキュメントで確認する：
`[言語名] [テストフレームワーク名] best practices [現在年]`

### 11. コード品質（Code Quality）

「どれくらいの品質で作るか」は非機能要件として定義する。
定義しないとAIも人間も「動けばいい」を基準にしてしまう。
→ 詳細は `.opencode/standards/principles/code-quality.md` を参照。

```
品質基準：
  [ ] 1ファイルの最大行数：file-size-and-cohesion.md の閾値（300行）
  [ ] テストカバレッジ目標：ビジネスロジックの ___% 以上
```

## AIへの指示時の活用方法

非機能要件を定義しておくことで、AIへの指示が具体的になる。

```
# 曖昧な指示（AIは「動けばよいコード」を生成する）
「株式一覧を取得するAPIを作って」

# 非機能要件を含む指示（AIは適切な設計を選択できる）
「株式一覧を取得するAPIを作って。
 要件：
 - p95で200ms以内に応答する
 - 最大10000件のデータを扱う
 - 1回あたり最大50件をページネーションで返す
 - 認証済みユーザーのみアクセス可能
 - すべてのリクエストをINFOレベルでログ出力する」
```
