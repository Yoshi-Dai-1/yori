# ADR 003: 知識・雛形の管理を三層構造にする

**日付**：2025年  
**状態**：採用

---

## 状況

プロジェクトをまたいでベストプラクティスを再利用するために、以下の選択肢を検討した：

1. GitHubテンプレートリポジトリのみ
2. AIエージェントスキルのみ
3. ナレッジベース（このリポジトリ）のみ
4. 複数の組み合わせ

## 決定

三層構造で管理する：

```
Layer 1: yori（このリポジトリ）
         思想・判断基準・命名規則を記録するナレッジベース

Layer 2: GitHubテンプレートリポジトリ（種別ごと）
         動作検証済みの雛形コード

Layer 3: 各プロジェクトの .opencode/ ディレクトリ
         プロジェクト固有の文脈をAIに渡すファイル群
```

## 理由

**ナレッジベースを最優先にする理由**：
- テンプレートは使わずに古くなることがある
- 「なぜその構成か」という判断基準は、プロジェクトの種別を問わず普遍的に使える
- AIへの指示時に渡せる文脈として機能する

**テンプレートリポジトリを分ける理由**：
- 動作確認済みのコードと思想の記録を混在させない
- テンプレートは「実際に使ったもの」だけを入れる（使われないテンプレートは作らない）

**AIスキルを最後にする理由**：
- ナレッジベースとテンプレートが安定してから、それらを参照するスキルを作る
- 土台なきスキルは再現性がない

## 運用フロー

```
新プロジェクト開始時：
  1. yori を参照して構成を決定
  2. 最も近いテンプレートリポジトリから新リポジトリを作成
  3. .opencode/project-context.md を書く
  4. 開発中の判断を decisions/ にフィードバック
```

## 優先順位

今すぐ着手すべき順序：
1. yori の整備（このリポジトリ）
2. 次の実プロジェクトに対応するテンプレートリポジトリを1つ作成
3. AIスキルはテンプレートが安定してから
