# TDD Cycle - 詳細手順

このルールはテストファイル編集時に Plugin が注入する。
AGENTS.md の TDD Cycle セクションを補完する詳細手順。

---

## テストの品質フレームワーク

テストコードは以下の指針に従う。これらは競合せず、**テスト構造を AAA Pattern で書き、テストスイート全体の品質を FIRST で担保する**。

### AAA Pattern（テスト構造）

`.opencode/standards/principles/code-review.md` を参照。
1つのテストメソッドを **Arrange（準備）→ Act（実行）→ Assert（検証）** の3セクションに分ける。

### FIRST 原則（テストスイート品質）

| 原則 | 意味 | 違反例 |
|------|------|--------|
| **F**ast | 単体テストはミリ秒単位で完了する | DBやHTTP通信を含む |
| **I**solated | テストは他のテストに依存しない。実行順序に依存しない | 共有状態（static変数・DBレコード）に依存 |
| **R**epeatable | 何度実行しても同じ結果になる | 日付・乱数・外部APIの現在値に依存 |
| **S**elf-validating | 試験結果は人間の解釈なしに pass/fail が判定できる | ログを目視確認するテスト |
| **T**imely | テストは実装の前に書く（TDD）か、実装と同時に書く | 実装完了後にテストを後付けする |

## テストコードの生成（TDDサイクルのStep2）

`@test-generator` を呼び出す。
テストの種類・優先順位・命名規則は `.opencode/standards/principles/test-strategy.md` および `.opencode/standards/principles/naming-conventions.md` を参照すること。ユニットテストを基本とし、必要な場合のみ結合テスト・E2Eテストを追加する。

## バグ修正時のルール

修正の前に、バグを再現するテストを先に書く。
`@test-generator` を呼び出してテストコードを生成してから実装に入る。
再現手順は最小限に絞り込んでからテストコードに変換する（関連しない条件を取り除き、バグの本質だけを再現する）。

## テストドリフト検知

ソースファイルが編集され、同一セッションで対応するテストファイルが編集されない場合：

1. 編集されたソースファイルの公開関数/クラス一覧と対応テストファイルのカバレッジを比較する
2. テストされていない変更点をリストアップし、`@test-generator` の呼び出しを提案する
3. **除外条件**: 編集がコメント・空白・型定義のみの変更、またはテストファイルが存在しないことが意図的（スクリプト・自動生成ファイル）の場合
