# AI協働開発におけるTDD（テスト駆動開発）

## なぜAI協働開発にTDDが有効か

AIはコードを高速に生成するが「動くかどうか」を確認せずに書く。
仕様を自然言語から解釈してパターンマッチで出力するため、細部の整合性が崩れやすい。

TDDはAIが書いたコードを人間がレビューする前の**第一フィルター**として機能する。
テストが通るまでAIに修正を続けさせ、通ったときだけ人間がレビューする。
これにより人間のレビューコストを大幅に削減できる。

---

## AIとのTDDが通常のTDDと異なる点

通常のTDD（Red → Green → Refactor）をそのままAIに指示しても機能しない。
AIは「テストを先に書いて」と言われると、テストと実装を同時に「それっぽく」書く。
結果、テストが実装に合わせて書かれた（テストの意味がない）状態になる。

**正しい手順：**

```
Step 1. 人間がテストケースを日本語で定義する     ← 最重要
Step 2. AIがテストコードに変換する（実装は書かない）
Step 3. 人間がテストコードを確認・修正する       ← 省略禁止
Step 4. AIがテストを通す実装を書く
Step 5. テストを実行して結果を確認する
Step 6. 失敗した場合はAIに修正を依頼し Step 5 に戻る
Step 7. 通過したら人間がコードをレビューする
Step 8. AIがコミットする（AGENTS.md の「コミット実行」設定に従う）
```

Step 3が最重要。**テストが正しいかを人間が確認する**。
これを省略すると「間違った実装を間違ったテストで通す」無意味な状態になる。

人間がコードを読めない場合のStep 3：
AIはテストコードを日本語で説明し、人間が期待との一致を確認する。
確認が取れたらStep4へ。修正が必要なら指示を受けて修正する。

## Step 1：テストケースの定義方法

テストケースは以下の構造で日本語で書く。

```
関数名・機能名：formatCurrency
テストケース：
  正常系：
  - 1000 を渡すと "¥1,000" を返す
  - 0 を渡すと "¥0" を返す
  - 1000000 を渡すと "¥1,000,000" を返す

  エッジケース：
  - 負の数（-1000）を渡すと "-¥1,000" を返す
  - 小数（100.5）を渡すと "¥100" を返す（小数点以下切り捨て）

  異常系：
  - null を渡すとエラーをスローする
  - 文字列（"abc"）を渡すとエラーをスローする
```

エッジケースと異常系を人間が定義することが、AIの見落としを補う核心。

## TDDが有効なケース・有効でないケース

| ケース | TDD有効か | 理由 |
|--------|----------|------|
| 純粋関数（計算・変換・バリデーション） | ✅ 高い | 入力と出力が明確で書きやすい |
| ビジネスロジック（サービス層） | ✅ 高い | 仕様の誤解釈を防げる |
| APIエンドポイント | ✅ 中程度 | 統合テストとして有効 |
| UIコンポーネント | ⚠️ 低い | レンダリングの確認が難しい |
| 外部API連携 | ⚠️ 低い | モックが実際のAPIと乖離しやすい |
| 探索的な実装（何を作るか未確定） | ❌ 不向き | 仕様が固まってから適用する |

UIとデータ取得が混在するフロントエンドでは、ビジネスロジックをhooksやservicesに切り出してからTDDを適用するのが現実的。

## TDDと他のツールの組み合わせ

TDDだけでは解決しないエラーがある。組み合わせで補う。

```
型エラー（TypeScript）       → tsc --noEmit を CI で実行
アーキテクチャ違反           → ESLint import/no-cycle
コードスタイルの不統一       → ESLint + Prettier
セキュリティの脆弱性         → npm audit
外部APIの仕様変更            → E2Eテスト（Playwright等）
```

理想の確認順序：

```
AIがコードを出力
  → tsc（型チェック）         ← 最速・ゼロコスト
  → ESLint（ルール確認）      ← 自動修正可能
  → ユニットテスト実行         ← TDDの核心
  → @code-reviewer            ← 独立コンテキストでのレビュー（自己評価バイアス排除）
  → 人間によるレビュー         ← ここで初めて人間が見る
  → E2Eテスト（必要な場合）
```

この順序は `principles/code-quality.md` でも参照される唯一の定義（SSOT）。

## TDD定着（.opencode/coding-conventions.md への追記）

初回セットアップ時、AIは以下を `.opencode/coding-conventions.md` のテスト方針として追記する：

```markdown
## テスト方針

- ビジネスロジック（services・hooks・utils）はTDDで実装する
- テストケースは人間が日本語で定義し、AIがコードに変換する
- テストコードは実装コードより先に書く
- AIが実装を書く際、テストコードは変更しない
- テストが通らない実装は提出しない（エラーメッセージとともに修正を継続する）
```

## テストの通過をどう判断するか

AIエージェントはテスト実行後に結果を報告する。

**通過したとき（Green）：**
```
✓ filterByPER([{code:"1234", per:10}], 15) → [{code:"1234", per:10}]  通過
✓ filterByPER([{code:"1234", per:16}], 15) → []  通過
✓ filterByPER([{code:"1234", per:null}], 15) → []  通過

3件のテスト、すべて通過しました。
```

**失敗したとき（Red）：**
```
✓ filterByPER([{code:"1234", per:10}], 15) → [{code:"1234", per:10}]  通過
✗ filterByPER([{code:"1234", per:null}], 15) → []  失敗
  期待値: []（空の配列）
  実際の値: [{code:"1234", per:null}]（nullが除外されていない）

1件失敗。nullのハンドリングを修正します。
```

テスト結果の報告形式（AIが守ること）：
- 期待値と実際の値を明確に表示する
- 通過/失敗を明示する
- 失敗時は「何が期待と違ったか」を日本語で説明する
- テスト自体が間違っている可能性があることを認識し、人間に最終確認を促す
