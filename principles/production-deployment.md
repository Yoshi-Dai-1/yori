
## 1. デプロイ先の選定

### 選択肢と判断基準

| サービス | 向いているケース | 月額コスト目安 |
|---------|----------------|-------------|
| Vercel | フロントエンド・Next.js | 無料〜（使用量次第） |
| Fly.io | バックエンド・フルスタック | 無料〜$5程度 |
| Render | バックエンド・DB込み | 無料〜$7程度 |
| Railway | バックエンド・手軽さ重視 | $5〜 |
| AWS / GCP | 大規模・細かい制御が必要 | 従量課金（複雑） |
| GitHub Pages | 静的サイトのみ | 無料 |

個人開発の初回本番リリースには Vercel（フロント）+ Fly.io または Render（バック）が現実的。

決定したら `decisions/` に `tech-decision.md.template` で記録する。

### 環境の分離

```
開発環境  → ローカルPC・開発用DB
ステージング環境 → 本番と同じ構成・本番前の最終確認（任意だが推奨）
本番環境  → 実際のユーザーが使う
```

最低限「開発」と「本番」を分離する。同じDBを共有しない。

## 2. The Twelve-Factor App 適用

本番移行時に適用する因子を優先順に示す。
各因子の詳細はAIに「第N因子を適用したい。ARCHITECTURE.mdを踏まえて何をすべきか教えて」と質問する。

### 優先度：高（リリース前に必ず対応）

**第3因子：設定を環境変数で管理**
コード内に設定値・APIキー・DB接続文字列を直書きしない。
`.env.example` が最新であることを確認する。
→ すでに `ssot-and-constants.md` でカバー済み。

**第5因子：ビルド・リリース・実行の分離**
「コードをビルドする」「本番環境に配置する」「サービスを起動する」を別々のステップにする。
GitHub Actions 等のCI/CDで自動化する。

**第10因子：開発・本番環境の一致**
「ローカルでは動くが本番で動かない」を防ぐ。
Dockerを使うか、同じNode/Pythonバージョンを使うことで差異を最小化する。

### 優先度：中（リリース後に対応可）

**第6因子：プロセスをステートレスに保つ**
サーバーのメモリにセッション情報を持たせない。
セッションはDB・Redisなど外部に保存する。
→ スケーリングが必要になったとき・複数サーバーに増やすときに重要。

**第11因子：ログをイベントストリームとして扱う**
ログをファイルに書き出すのではなく、標準出力（stdout）に書き出す。
ログの収集・保存はインフラ（Fly.io・Vercel等）に任せる。

### 優先度：低（必要になったときに対応）

**第8因子：並行性（プロセスモデルでスケール）**
→ ユーザーが増えてスケーリングが必要になったとき。

**第9因子：廃棄容易性（高速起動・グレースフルシャットダウン）**
→ デプロイ頻度が高くなったとき。

残りの因子（I・II・IV・VII・XII）は現時点でほぼ自動的に満たされているか、
対応が不要なケースが多い。

## 3. CI/CDパイプライン

最小構成（GitHub Actions）：

```yaml
# .github/workflows/ci.yml の骨格
on: [push, pull_request]
jobs:
  test:
    - tsc（型チェック）
    - ESLint
    - ユニットテスト
  deploy:
    needs: test
    if: branch == main
    - デプロイ先へのデプロイコマンド
```

「テストが通らないとデプロイできない」状態を自動で強制する。

**CI/CDパイプラインの設計原則：**

```
環境の分離：
  dev（開発）→ staging（検証）→ production（本番）の順でデプロイする
  本番への直接デプロイは禁止する

デプロイ戦略（プロジェクト規模に応じて選択する）：
  ローリングデプロイ   → 順次置き換える。シンプルだがロールバックに時間がかかる（小〜中規模向け）
  ブルーグリーンデプロイ → 新旧環境を並行稼働しトラフィックを切り替える。即時ロールバック可能（中〜大規模向け）
  カナリアリリース    → 一部のトラフィックのみ新バージョンに流す。段階的な検証が可能（大規模・商用向け）

ロールバック設計：
  本番デプロイ前にロールバック手順を必ず定義する
  「デプロイしたら戻せない」状態でリリースしない
```

具体的なパイプライン実装はツールの公式ドキュメントで確認する：
`[CI/CDツール名] deployment pipeline best practices [現在年]`
（GitHub Actions / GitLab CI / CircleCI / Cloud Build 等）

詳細はAIに「[CI/CDツール名]で[デプロイ先]へのCI/CDを構築して」と依頼する。

## 4. シークレット管理

本番環境の機密情報（APIキー・DB接続文字列）は：
- コードに書かない（`.gitignore` で `.env` を除外済みであること）
- GitHub ActionsのSecrets・デプロイ先のEnvironment Variablesで管理する
- ローカルの `.env` と本番の環境変数は別の値を使う

## DDDの適用判断

このファイルに記載している理由：本番移行時にビジネスロジックの複雑さが顕在化するため。

### 役割分担

**人間がすること：体感を言葉にする**
「最近、条件を変えるたびにあちこち修正が必要になってきた気がする」
「同じようなチェックをいろんな場所でやっている気がする」
この体感をAIに伝えるだけでよい。コードを読む必要はない。

**AIがすること：コードを診断して提案する**
以下の指示でAIが自律的に診断する：

```
現在のコードを読んで、DDDを適用すべき症状が出ていないか診断してください。
確認する症状：
  症状1：同じバリデーション・ビジネスルールが複数箇所に重複している
  症状2：ビジネスルールの意図がコードから読み取りにくくなっている
  症状3：条件を1つ変えると複数ファイルの修正が必要になっている

該当する症状があれば、どのコードに・どのDDD概念を・どう適用するかを提案してください。
該当しなければ「現時点では不要」と明示してください。
```

**人間が判断すること：AIの提案を採用するかどうか**
AIが「症状1に該当します。ValueObjectを適用することを提案します」と言ったとき、
「やって」か「今はやらない」かを決めるのは人間。コードは読まなくてよい。

### live-operation Skill への組み込み

運用フェーズでは `.claude/skills/live-operation/` のPeriodic Diagnosisとして、
以下を定期的にAIに実行させることができる：

```
現在のビジネスロジック（services/・features/）を読んで、
DDDの適用が有効になる症状が出ていないか診断してください。
```

これを月1回・または「なんか面倒になってきた」と感じたタイミングで実行する。

### DDDの主要概念（AIへの指示で使う用語）

**ValueObject（値オブジェクト）**
ビジネスルールを持つ不変の値。
例：PERという数値は「0以上・1000以下でなければならない」制約を持つ。
この制約をPERを使う側全員に書かせるのではなく、PER型の中に定義する。
→ AIへの指示：「PERをValueObjectとして実装して」

**DomainService（ドメインサービス）**
複数の値にまたがるビジネスルール。
例：「スクリーニング条件を評価する」は特定の銘柄でも条件でもなく、その組み合わせの処理。
→ AIへの指示：「スクリーニング評価をDomainServiceとして実装して」
