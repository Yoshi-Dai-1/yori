# レジリエンス（Resilience）：壊れても死なない設計

## なぜ「壊れる前提」で設計するのか

完全に壊れないシステムは存在しない。
「壊れないようにする」と「壊れても回復できるようにする」は別の設計。
両方必要だが、後者が抜けているプロジェクトが多い。

壊れる原因は技術的なものだけではない：
- セキュリティ侵害（外部攻撃・未知の脆弱性）
- インフラ障害（クラウドの障害・ネットワーク切断）
- オペレーションミス（自分のミス・誤ったマイグレーション）
- 依存サービスの終了（外部APIの廃止・ライブラリのメンテナンス放棄）
- データの破損・消失（ストレージ障害・ソフトウェアバグ）

**「壊れた後に何が起きるか」を設計しないプロジェクトは、
 壊れたときに初めて設計を始める。その時点で手遅れになることがある。**

## 4つの防御層

### Layer 1：壊れにくくする（Prevention）

→ セキュリティ（security-implementation.md）・TDD（tdd-with-ai.md）・
  コードレビュー（code-review.md）で対応済み。
→ 外部通信の安定性設計（network-resilience.md）もこの層に属する。
  タイムアウト・リトライ・冪等性の設計は「壊れにくくする」防御層の一部。
  rules/network-resilience.md が通信コードの編集時に自律的に確認する。

### Layer 2：壊れたことに気づく（Detection）

「気づかないまま壊れ続ける」が最もコストが高い状態。

```
実装すべき検知の仕組み：

エラー検知：
  [ ] Sentry等のエラー監視ツールで本番エラーをリアルタイム通知
  [ ] ERRORレベルのログがSlack・メールに通知される

死活監視：
  [ ] UptimeRobot等でサービスのダウンを検知（無料枠あり）
  [ ] /health エンドポイントを実装してDBへの疎通も確認する

異常検知：
  [ ] レートリミット超過のアラート
  [ ] 異常に大きいリクエスト・レスポンスの検知
  [ ] 深夜・休日の大量アクセスの通知
```

### Layer 3：被害を限定する（Containment）

1点突破で全体が落ちない設計。

```
最小権限の実装：
  [ ] DBユーザーはアプリが必要な権限のみ（READ/WRITEのみ。DROP禁止）
  [ ] 外部APIキーのスコープを必要最小限に設定する
  [ ] 本番と開発のDBを完全に分離する（接続文字列・認証情報ともに別）

データ保護：
  [ ] 機密データはDBレベルで暗号化（DBが漏洩しても平文が読めない）
  [ ] バックアップも暗号化する
  [ ] ログに個人情報・機密情報を出力しない

環境の分離：
  [ ] 本番・ステージング・開発の環境が完全に分離されている
  [ ] 開発環境から本番DBに接続できない構成
```

### Layer 4：回復する（Recovery）

壊れた後に何ができるかを事前に設計する。

```
バックアップ：
  [ ] バックアップが設定されているか
  [ ] バックアップの頻度が決まっているか（RPOの定義）
  [ ] バックアップから実際にリストアできることを確認したか（★最重要）

復旧手順：
  [ ] ロールバック手順が文書化されているか（docs/operations.md）
  [ ] デプロイ失敗時の手順が決まっているか
  [ ] DBマイグレーション失敗時の手順が決まっているか

インシデント対応：
  [ ] 何が起きたかを記録する手順があるか
  [ ] ユーザーへの通知基準が定義されているか（情報漏洩時の開示義務）
```

## バックアップの原則：3-2-1ルール

```
3：データのコピーを3つ保持する
2：2種類の異なるメディア・場所に保存する
1：1つはオフサイト（別リージョン・別サービス）に置く
```

個人開発での現実的な適用：
- 本番DB（クラウド）
- クラウドの自動バックアップ（別リージョン）
- 定期的なエクスポート（S3・ローカル）

### バックアップで最も重要なこと

**「バックアップがある」と「バックアップから復元できる」は別のこと。**
バックアップ設定完了後、最初に1回リストアテストを実施する。以降は月1回実施する。リストアテストの手順と結果（成功/失敗・所要時間）を `docs/operations.md` に記録する。

```
月1回実施すること：
  1. バックアップからステージング環境にリストアする
  2. データが正常に復元されているか確認する
  3. リストアにかかった時間を記録する（RTO の実測値として使う）

これをしないバックアップは存在しないと同じ。
```

## 依存関係のレジリエンス

外部サービス・ライブラリが突然使えなくなることを想定する。

### クリティカルな依存関係を把握する

```markdown
<!-- ARCHITECTURE.md の「依存関係リスク」セクションに記録する -->

## 依存関係リスク

| 依存先 | 用途 | これが止まると | 代替手段 | 移行コスト |
|--------|------|--------------|---------|-----------|
| [外部APIサービス名] | [用途] | [影響] | [代替] | [コスト] |
| [ライブラリ名] | [用途] | [影響] | [代替] | [コスト] |
```

### 依存関係の健全性維持

```bash
# 月次で実行する（または CIに組み込む）
npm audit                    # 既知の脆弱性チェック
npm outdated                 # 更新可能なパッケージの確認
npx depcheck                 # 使われていない依存関係の検出

# Python の場合
pip-audit                    # 既知の脆弱性チェック
pip list --outdated          # 更新可能なパッケージの確認
```

## ヘルスチェックエンドポイントの実装

死活監視が機能するためには、アプリケーション側の実装が必要。

```typescript
// api/routes/health.routes.ts
// GET /health → システムの健全性を返す

router.get('/health', async (req, res) => {
  const checks = {
    status: 'ok',
    timestamp: new Date().toISOString(),
    uptime: process.uptime(),
    db: 'unknown',
  };

  try {
    // DBへの疎通確認（実際のクエリで確認）
    await db.query('SELECT 1');
    checks.db = 'ok';
  } catch (error) {
    checks.db = 'error';
    checks.status = 'degraded';
  }

  const statusCode = checks.status === 'ok' ? 200 : 503;
  res.status(statusCode).json(checks);
});
```

## 運用手順書（docs/operations.md）のテンプレート

レジリエンスの実装の最終形は「文書化された手順」。
以下をプロジェクトの `docs/operations.md` に記録する。

```markdown
# 運用手順書

## 日常的な運用

### デプロイ方法
[手順を書く]

### ロールバック方法
[手順を書く。git revert [コミットID] など]

### DBバックアップの確認方法
[手順を書く]

### バックアップからのリストア手順（月1回実施）
[手順を書く。最後の実施日を記録する]

## 障害対応

### サービスがダウンしたとき
1. [確認すること：ログ・モニタリング]
2. [対処すること：再起動・ロールバック]

### データが壊れたとき
1. [確認すること]
2. [復旧手順：バックアップからのリストア]

### セキュリティインシデントが発生したとき
1. [即座にやること：影響範囲の特定・アクセス遮断]
2. [ユーザーへの通知基準と方法]
3. [記録すること：何が・いつ・どう起きたか]

## 月次チェックリスト

[ ] バックアップからのリストアテスト実施
[ ] npm audit / pip-audit 実行・脆弱性対応
[ ] 依存ライブラリの更新確認
[ ] @resilience-checker を実行して診断
[ ] モニタリング・アラートが正常に動作しているか確認
```

## AIへの活用方法

### 新機能実装時

```
この機能（[機能名]）を実装するにあたり、
resilience.md の観点から設計を確認してください：

- この機能が動かなくなった場合の最大の影響は何か
- 外部APIへの依存があれば、network-resilience.md の判断手順で
  タイムアウト・リトライ・冪等性・サーキットブレーカーの採否を判定して
- この機能のバックアップ・復旧を考慮すべき点はあるか
```

### 月次の定期診断

```
principles/resilience.md と ARCHITECTURE.md を参照して、
現在のシステムのレジリエンスを評価してください。

確認してほしい項目：
1. バックアップのリストアテストが最近実施されているか（docs/operations.md確認）
2. 依存関係リスクのリストが最新か（ARCHITECTURE.md確認）
3. ヘルスチェックエンドポイントが機能しているか
4. 直近1ヶ月のエラーログで見落とされているパターンがないか

GREEN / YELLOW / RED で総合評価してください。
```
