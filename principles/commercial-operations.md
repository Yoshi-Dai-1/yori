# 商用運用ガイド（Commercial Operations）

## このファイルの目的

個人・小規模プロジェクトと商用プロジェクトでは、要求される水準が構造的に異なる。
このファイルは商用プロジェクト固有の要件を定義し、個人開発との差分を明確にする。

「商用」の定義：ユーザーへのサービス提供・収益・SLA・チーム開発のいずれかを伴うプロジェクト。

---

## 個人開発と商用の主な差分

| 項目 | 個人・小規模 | 商用 |
|------|------------|------|
| 可用性目標 | 99%（年87時間停止許容） | 99.9〜99.99%（SLA定義が必要） |
| リリース | 手動・随時 | 変更管理・承認フロー・計画的リリース |
| コード変更 | 個人の判断で即マージ | PR必須・レビュー承認・CODEOWNERS |
| インシデント | 気づいたら対応 | 検知→対応→報告→ポストモーテム |
| セキュリティ | チェックリスト | 定期監査・ペネトレーションテスト |
| 監査 | 不要なことが多い | アクセスログ・操作ログの長期保存 |
| ブランチ | main直接コミット可 | ブランチ戦略の定義が必要 |
| コスト | 個人負担・小額 | 予算管理・コスト配分・SLA違反時の補償 |

## 1. SLA / SLO / SLI の定義

商用サービスでは「どのくらい動くか」を数値で約束する。

```
SLI（Service Level Indicator）：実際に計測する指標
  例：過去30日間の成功リクエスト率 = 成功件数 / 全リクエスト件数

SLO（Service Level Objective）：達成目標（内部目標）
  例：成功リクエスト率 99.9% 以上を維持する

SLA（Service Level Agreement）：顧客との契約上の合意
  例：月間稼働率 99.9% 未満の場合、翌月の料金を○%クレジット
```

SLO は SLA より厳しく設定する（SLO 99.95% → SLA 99.9%）。
理由：SLO に余裕を持たせることで、SLA 違反前に気づいて対処できる。

**定義テンプレート：**

```markdown
## SLI / SLO / SLA

### SLI（計測指標）
- 可用性：成功レスポンス率（2xx/3xx ÷ 全リクエスト）
- レイテンシ：p95 レスポンスタイム
- エラー率：5xxエラー ÷ 全リクエスト

### SLO（内部目標）
- 可用性：99.95% / 月
- レイテンシ p95：500ms 以内
- エラー率：0.1% 未満

### SLA（顧客への約束）
- 月間稼働率：99.9%
- 違反時の対応：[クレジット / 返金 / 通知] を定義する

### エラーバジェット
SLO 99.95% = 月間 21.6分の停止を許容
このバジェットを使い切る前にリリース頻度・変更規模を調整する
```

## 2. ブランチ戦略

チーム開発では「誰が・いつ・何をマージするか」を明示的に定義する。
定義のないブランチ運用はコンフリクトと意図しないリリースを生む。

### プロジェクト規模による選択

AIは `docs/project-definition.md` のチーム規模を読んで以下を適用する:
チーム規模の値を以下の基準で判定する:
「1」「一人」「1人」「personal」「solo」「個人」→ 個人（1人）扱い
「2」「3」「4」「5」または「2人」「小規模」「small」→ 2〜5人扱い
「6」以上または「中規模」「大規模」「enterprise」→ 6人以上扱い
いずれにもマッチしない場合は人間に確認する

| チーム規模 | 適用する戦略 |
|-----------|------------|
| 個人（1人） | シンプル戦略（以下参照） |
| 2〜5人 | Trunk-Based Development（以下参照） |
| 6人以上 または リリースサイクルが月1回以下 | GitFlow（以下参照） |

docs/project-definition.md にチーム規模の記載がない場合は人間に確認する。

### 個人開発：シンプル戦略

```
main（本番ブランチ）
  ↑
feature/xxx  → 数日以内にマージ（1週間を超えたら設計を見直す）
fix/yyy      → 即日マージ推奨
```

ルール（3つのみ・これ以外のルールを追加しない）：
1. main は常にデプロイ可能な状態に保つ（壊れたコードを直接 push しない）
2. 機能追加・バグ修正は必ずブランチを切る（main への直接 push は設定ファイルの変更・typo 修正のみ許可）
3. マージ後はブランチを即座に削除する

ブランチプロテクション：設定任意。CI が通ったことを確認してから自分でマージする。

コミットメッセージ規約：`.opencode/standards/principles/naming-conventions.md` の
「コミットメッセージ（Conventional Commits）」セクションを参照する。
型を使う理由：CHANGELOG 自動生成と SemVer の自動判定が可能になる。

### 推奨：Trunk-Based Development（小〜中規模チーム）

```
main（trunk）
  ↑
feature/xxx  → 短命（1〜3日以内）
fix/yyy      → 短命（即日マージ推奨）
```

- 長命なブランチを作らない（2週間以上のブランチは設計の問題）
- フィーチャーフラグで未完成機能を隠しながら main にマージする
- CI が通らなければ main にマージできない（ブランチプロテクション必須）

### 代替：GitFlow（リリースサイクルが長い場合）

```
main         本番リリース済みのコード
develop      次のリリースに向けた統合ブランチ
feature/xxx  機能開発
release/1.2  リリース準備
hotfix/yyy   本番緊急修正
```

GitFlow は複雑なため、チームが小さい場合は Trunk-Based Development を推奨。

### ブランチプロテクションルール（必須設定）

```
main ブランチ：
  [ ] 直接 push 禁止（PR 必須）
  [ ] PR には最低 1 人の承認が必要
  [ ] CI（テスト・lint）が通らなければマージ禁止
  [ ] 古いブランチは自動的に削除
```

## 3. プルリクエスト・コードレビューのガバナンス

### CODEOWNERS の設定

重要なファイル・ディレクトリには自動的にレビュアーを割り当てる。

```
# .github/CODEOWNERS

# 認証・セキュリティ関連は必ずセキュリティ担当がレビュー
/src/api/auth/          @security-team
/src/services/auth*     @security-team

# インフラ・CI/CD の変更はインフラ担当が必須
/.github/               @infra-team
/docker/                @infra-team

# ビジネスロジックはドメイン担当が必須
/src/services/billing*  @backend-lead
```

### PRテンプレート（`.github/pull_request_template.md`）

```markdown
## 変更の概要

## 変更の種類
- [ ] バグ修正
- [ ] 新機能
- [ ] リファクタリング
- [ ] パフォーマンス改善
- [ ] セキュリティ修正
- [ ] ドキュメント

## テスト
- [ ] 既存テストが通ることを確認した
- [ ] 新しいテストを追加した（バグ修正・新機能の場合）
- [ ] ローカルで動作確認した

## セキュリティ
- [ ] 機密情報がコードに含まれていない
- [ ] 入力バリデーションが実装されている
- [ ] 認証・認可が正しく機能している

## 本番影響
- [ ] DBマイグレーションが含まれる場合、ロールバック手順を確認した
- [ ] ダウンタイムが発生する変更の場合、メンテナンス通知を計画した
```

## 4. 変更管理・リリース戦略

### リリース種別と判断基準

```
緊急 Hotfix（本番障害・セキュリティ脆弱性）
  → 通常フローを省略して即時リリース可
  → リリース後に必ずポストモーテムを実施

通常リリース
  → ブランチプロテクション + PR承認 + CI通過 が必須
  → ステージング環境での確認後にリリース

大規模リリース（DBスキーマ変更・APIバージョニング等）
  → リリース計画書を decisions/ に記録する
  → ロールバック手順を事前に文書化する
  → カナリアリリースまたはフィーチャーフラグを使用する
```

### カナリアリリース（推奨）

新機能を全ユーザーに一度に展開せず、段階的に広げる。

```
5% のユーザーに展開 → エラー率・パフォーマンスを監視
  → 問題なければ 20% → 50% → 100%
  → 問題があれば即座に 0% に戻す（フィーチャーフラグで制御）
```

### フィーチャーフラグ

未完成・リスクのある機能を main にマージしながら、本番では無効にする。

```typescript
// 未完成の機能をフラグで制御
if (featureFlags.isEnabled('new-screener-v2', userId)) {
  return <NewScreenerV2 />;
}
return <ScreenerV1 />;
```

使用条件：長期ブランチを避けるため・ABテストのため・緊急時の即時無効化のため。

## 5. インシデント管理

### インシデントの定義と重要度

```
P0（Critical）：サービス全停止・データ損失・セキュリティ侵害
  対応開始：15分以内
  解決目標：2時間以内
  通知：経営層・顧客（SLA違反の場合）

P1（High）：主要機能の障害・パフォーマンス重大劣化
  対応開始：30分以内
  解決目標：4時間以内
  通知：チームリーダー

P2（Medium）：一部機能の障害・断続的なエラー
  対応開始：2時間以内
  解決目標：翌営業日
  通知：担当チーム

P3（Low）：軽微な不具合・UXの問題
  → 通常の開発フローで対応
```

### インシデント対応フロー

```
1. 検知（監視アラート / ユーザー報告）
2. 初動確認（誰が対応するか決める・重要度を判定する）
3. 影響範囲の特定（何が・どのくらい・誰に影響しているか）
4. 暫定対応（ロールバック / フィーチャーフラグ無効化 / トラフィック遮断）
5. 根本原因の調査
6. 恒久対応（修正リリース）
7. ポストモーテム（P0・P1の場合は必須）
8. 予防策の実装
```

### ポストモーテム（事後検証）テンプレート

非難なき文化（Blameless）が前提。個人を責めず、システムと仕組みを改善する。

```markdown
# ポストモーテム [YYYY-MM-DD]

## インシデント概要
- 重要度：P[0/1/2]
- 発生日時：
- 検知日時：
- 解決日時：
- 影響ユーザー数：
- ダウンタイム：

## タイムライン
| 時刻 | 出来事 |
|------|--------|
| XX:XX | アラート発火 |
| XX:XX | 対応開始 |

## 根本原因
[何が・なぜ起きたか。個人ではなくシステムの問題として記述]

## 暫定対応
[何をして止めたか]

## 恒久対応
[再発防止のために何を変えるか]

## 改善アクション
| アクション | 担当 | 期限 | 完了 |
|-----------|------|------|------|
| [具体的な改善内容] | | | [ ] |

## 学んだこと
[チームとして何を学んだか]
```

## 6. 監査ログ

商用サービスでは「誰が・いつ・何をしたか」を記録・保存する義務が生じることがある。

### 監査ログが必要な操作

```
必ず記録すべき操作：
  - ログイン成功・失敗（不審なアクセスの検知）
  - パスワード変更・アカウント削除
  - 権限の付与・剥奪（管理者操作）
  - 個人情報へのアクセス・変更（個人情報保護法対応）
  - 課金・決済操作
  - データのエクスポート・大量取得
  - システム設定の変更

記録する情報：
  - タイムスタンプ（ISO 8601・タイムゾーン付き）
  - 操作を行ったユーザーID
  - 操作の種類
  - 対象リソース（何を操作したか）
  - 送信元IPアドレス
  - 操作の結果（成功・失敗）
```

### 監査ログの保存・保護

```
保存期間：
  一般的な商用サービス：最低6ヶ月〜1年
  金融・医療系：法令に従う（例：電子帳簿保存法 7年）

保護方針：
  - 監査ログは改ざん不可にする（追記専用・ハッシュチェーン等）
  - アプリケーションログとは別ストレージに保存する
  - アクセス制限：監査ログ自体へのアクセスも記録する
```

### 実装テンプレート

```typescript
// shared/audit/audit-logger.ts
interface AuditLog {
  timestamp: string;       // ISO 8601
  userId: string;
  action: string;          // 'LOGIN' | 'DATA_EXPORT' | 'PERMISSION_CHANGE' | ...
  resource: string;        // 操作対象のリソース
  resourceId?: string;
  ipAddress: string;
  userAgent: string;
  result: 'SUCCESS' | 'FAILURE';
  details?: Record<string, unknown>; // 追加情報（個人情報は含めない）
}

// 使用例
await auditLogger.log({
  userId: session.userId,
  action: 'DATA_EXPORT',
  resource: 'stock_screener_results',
  ipAddress: req.ip,
  userAgent: req.headers['user-agent'],
  result: 'SUCCESS',
  details: { exportedRows: 1250 }
});
```

## 7. 商用セキュリティ要件

個人開発のチェックリストに加えて、以下を適用する。

### 定期セキュリティ監査

```
月次：
  [ ] 依存ライブラリの脆弱性スキャン（npm audit / pip-audit）
  [ ] アクセスログの異常パターン確認
  [ ] 権限の棚卸し（不要なアカウント・過剰な権限の削除）

四半期：
  [ ] 外部ペネトレーションテスト（商用サービスは外部委託を推奨）
  [ ] インフラのセキュリティ設定レビュー
  [ ] 機密情報の取り扱いレビュー（APIキーのローテーション）

年次：
  [ ] セキュリティポリシーの見直し
  [ ] コンプライアンス要件の変化への対応確認
```

### 個人情報・機密データの追加要件

```
商用サービスで個人情報を扱う場合（追加）：
  [ ] 個人情報の取り扱い方針を社内ドキュメントに記載する
  [ ] データ主体の権利に対応できる仕組み（削除・開示請求）
  [ ] データポータビリティ対応（GDPRが適用される場合）
  [ ] 個人情報の越境移転の制限を確認する
```

## AIへの活用方法

### プロジェクト開始時（商用と判明した場合）

```
このプロジェクトは商用サービスです。
principles/commercial-operations.md を読んで、
docs/project-definition.md の内容をもとに以下を評価してください：

1. SLA / SLO / SLI の定義が必要か（どの水準が妥当か）
2. ブランチ戦略として Trunk-Based と GitFlow のどちらが適切か
3. 監査ログの対象操作として何を記録すべきか
4. 定期セキュリティ監査のスコープを提案してください

評価後、ARCHITECTURE.md の非機能要件セクションに追記すべき内容を提案してください。
```

### インシデント発生時

```
P[0/1/2] インシデントが発生しました。
principles/commercial-operations.md のインシデント管理セクションに従って、
以下を支援してください：

1. 影響範囲の特定に必要な確認事項を列挙してください
2. 暫定対応の選択肢を提案してください（ロールバック / フィーチャーフラグ / 等）
3. docs/operations.md のインシデント対応手順を参照してください
```

### 月次の定期確認（商用）

```
@resilience-checker に加えて、以下を商用観点で確認してください：

principles/commercial-operations.md を参照して：
1. 今月のエラーバジェット消費量（SLO達成状況）
2. 監査ログが正常に記録されているか（サンプル確認）
3. アクセス権限の棚卸しが必要な項目はないか

結果を GREEN / YELLOW / RED で評価してください。
```
