# 本番運用を想定した設計チェックリスト（Production Readiness）

## このファイルの目的

「動く」と「本番で運用できる」は別物。
このチェックリストは、プロトタイプから本番リリースに移行するときに確認するもの。

開発中から意識しておくことで、リリース直前の大規模な作り直しを防ぐ。

---

## チェックリストの使い方

リリース前にこのリストを確認する。
「未対応」のまま進める場合は、理由をADRに記録する（意図的な妥協と技術的負債の区別のため）。

---

## 1. セキュリティ

詳細なセキュリティ監査は `@security-auditor`（監査モード）が担う。
このセクションはリリース前の最終確認チェックリストとして使用する。
AIが自律的に `@security-auditor`（監査モード）を呼び出した後、
このチェックリストで人間が最終確認する。

### 認証・認可
```
[ ] 認証が必要なエンドポイントはすべて保護されているか
[ ] 認証トークンの有効期限が設定されているか
[ ] ロールベースのアクセス制御が正しく機能しているか
[ ] 管理者機能へのアクセスが一般ユーザーに制限されているか
```

### データ保護
```
[ ] パスワードがハッシュ化されているか（平文保存は絶対禁止）
[ ] APIキー・シークレットが環境変数で管理されているか
[ ] コードリポジトリに機密情報がコミットされていないか（git-secretsなどで検証）
[ ] 本番DBへのアクセスが最小権限になっているか
[ ] 通信がHTTPS（TLS）で暗号化されているか
```

### 入力値・出力値
```
[ ] すべての外部入力がバックエンドでバリデーションされているか
[ ] エラーレスポンスにスタックトレースや内部情報が含まれていないか
[ ] SQLインジェクション対策がされているか（ORMのパラメータバインディングを使用）
[ ] XSS対策がされているか（ユーザー入力をエスケープせずにHTMLに出力していない）
[ ] CSRFトークンが実装されているか（フォーム送信がある場合）
```

### 依存ライブラリ
```
[ ] npm audit / pip audit でHIGH以上の脆弱性がないか
[ ] 使用していないパッケージが削除されているか
[ ] ライブラリのバージョンが固定されているか（package-lock.json / requirements.txt）
```

---

## 2. 信頼性・可用性

### エラーハンドリング
```
[ ] 想定外のエラーがユーザーに生のエラーメッセージを返していないか
[ ] すべての例外がキャッチされ、ログに記録されているか
[ ] 外部APIの呼び出しに接続タイムアウトと読み取りタイムアウトの両方が設定されているか
    （タイムアウト値が constants/ に定数として定義されているか）
[ ] ARCHITECTURE.md の「通信設計」セクションに採否と根拠が記録されているか
    （詳細な判断基準：.claude/standards/principles/network-resilience.md）
[ ] リトライを実装している場合、冪等でない操作（決済・メール送信・SMS送信・通知送信）が
    リトライ対象から除外されているか
[ ] 冪等でない操作にリトライが必要な場合、冪等性の確保が先に実装されているか
[ ] DB接続失敗時のフォールバック処理があるか（採用している場合のみ確認する）
```

### データ整合性
```
[ ] 重要な操作がトランザクションで保護されているか
[ ] DBのユニーク制約・NOT NULL制約が適切に設定されているか
[ ] データの二重送信を防ぐ仕組みがあるか（冪等性の確保）
```

### バックアップ
```
[ ] DBのバックアップが設定されているか
[ ] バックアップからの復元手順が文書化されているか
[ ] バックアップが実際に復元できることを確認したか
```

---

## 3. パフォーマンス

### フロントエンド
```
[ ] Core Web Vitals（LCP・FID・CLS）が目標値を満たしているか
[ ] 画像が最適化されているか（WebP形式・適切なサイズ）
[ ] 不要なJavaScriptが本番バンドルに含まれていないか
[ ] コード分割（Code Splitting）が適用されているか（大規模の場合）
[ ] 静的アセットにキャッシュヘッダーが設定されているか
```

### バックエンド・DB
```
[ ] 頻繁に使うクエリにインデックスが設定されているか
[ ] N+1問題が発生していないか（ORMのクエリログで確認）
[ ] 重いクエリにページネーションが実装されているか
[ ] キャッシュを実装している場合、キャッシュの無効化戦略が定義されているか
    キャッシュを実装していない場合はこの項目をスキップする
[ ] 本番相当のデータ量でパフォーマンスを検証したか
```

---

## 4. 監視・ログ

### ログ設計
```
[ ] ログレベルが適切に設定されているか（本番でDEBUGログが出ていないか）
[ ] ログにパスワード・APIキーが含まれていないか
[ ] エラーログに十分な情報（タイムスタンプ・コンテキスト・スタックトレース）があるか
[ ] ログの保存先・保存期間が決まっているか
```

### アラート・通知
```
[ ] エラーが発生したときに通知が来るか（メール・Slack等）
[ ] インフラのコスト上限アラートが設定されているか（クラウド利用の場合）
[ ] サービスのダウンを検知する仕組みがあるか
```

---

## 5. デプロイ・運用

### 環境管理
```
[ ] 開発・ステージング・本番の環境が分離されているか
[ ] 環境ごとに異なる設定が環境変数で管理されているか（コードで分岐していないか）
[ ] 本番DBに開発用データが入っていないか
```

### デプロイプロセス
```
[ ] デプロイ手順が文書化されているか（README or docs/）
[ ] デプロイに手動の秘伝のコマンドが必要な状態になっていないか
[ ] ロールバック手順が決まっているか（デプロイ後に問題が発覚した場合）
[ ] CIでテストが自動実行されているか（テストが通らないとデプロイできない）
```

### ドキュメント
```
[ ] セットアップ手順がREADMEに書かれているか（新メンバーが1人で環境構築できるか）
[ ] .env.exampleが最新の状態か
[ ] APIのエンドポイント一覧が文書化されているか
[ ] 既知の問題・制限事項が記録されているか
```

## 6. テスト

```
テスト戦略の確認：
  [ ] ARCHITECTURE.md の「テスト戦略」セクションに方針が記載されているか
  [ ] テストフレームワークが決定・設定されているか

テストの実装確認：
  [ ] ビジネスロジックの重要な部分にユニットテストがあるか
  [ ] 主要なAPIエンドポイントにインテグレーションテストがあるか
  [ ] テストが本番コードと同じリポジトリで管理されているか
  [ ] テストが自動実行されているか（CI/CD）
  [ ] テストカバレッジの目標が定義・達成されているか（ARCHITECTURE.mdに記載の目標値）
```

## 7. 法的・コンプライアンス

```
個人情報・プライバシー：
  [ ] プライバシーポリシーが作成・公開されているか（個人情報収集がある場合）
  [ ] 利用規約が作成・公開されているか（ユーザー向けサービスの場合）
  [ ] データの保存期間・削除ポリシーが定義されているか

ライセンス・外部サービス：
  [ ] 使用しているOSSライセンスを把握・遵守しているか
  [ ] 外部API・データソースの商用利用可否を確認しているか
  [ ] データソースの再配布・二次利用の可否を確認しているか

業界固有の規制：
  [ ] 金融情報を扱う場合：金融商品取引法への対応を確認したか
  [ ] 医療・介護情報を扱う場合：要配慮個人情報の管理を確認したか
  [ ] ユーザーの通信情報を外部送信する場合：電気通信事業法の外部送信規律に対応したか
  [ ] ペネトレーションテストを実施する場合：管理者の書面による許可を取得したか
  [ ] 政府・重要インフラ向けの場合：サイバーセキュリティ基本法・ISMAPへの対応を確認したか

アクセシビリティ（Webフロントエンドを含む場合）：
  [ ] ARCHITECTURE.md の「アクセシビリティ」セクションに適合レベル目標が記載されているか
  [ ] 定めた適合レベルのチェックリストを実施したか（non-functional-requirements.md 参照）
  [ ] 法的義務がある場合（公共機関等）：専門家による確認を実施したか

国際化（多言語展開がある場合）：
  [ ] ARCHITECTURE.md の「国際化」セクションに方針が記載されているか
  [ ] UIテキストがハードコードされていないか（翻訳ファイルで管理されているか）
  [ ] タイムゾーンがUTCで保存されているか

WebSocket（WebSocketを使用している場合）：
  [ ] 再接続戦略（指数バックオフ）が実装されているか
  [ ] Heartbeat（Ping/Pong）が実装されているか
  [ ] 本番環境でWebSocket接続時の認証・認可チェックが機能しているか

GraphQL（GraphQLを使用している場合）：
  [ ] 本番環境でIntrospectionが無効化されているか
  [ ] クエリの深さ・複雑度制限が設定されているか
  [ ] N+1問題への対策（DataLoader等）が実装されているか

データベースマイグレーション（DBスキーマ変更がある場合）：
  [ ] すべてのマイグレーションにdown（ロールバック）が定義されているか
  [ ] ゼロダウンタイムでのマイグレーションが可能か（本番稼働中の場合）
  [ ] マイグレーション失敗時のロールバック手順が `docs/operations.md` に記載されているか
```

## 8. 多層防御（侵入された後の備え）

「防ぎ切れなかった場合」を前提にした設計が、未知の脅威への唯一の対策。
詳細なチェックリストと実装方法は `principles/resilience.md` の Layer2〜4 を参照。

本番リリース前に確認すること：
- `principles/resilience.md` の「4つの防御層」チェックリストを実施済みか
- `docs/operations.md` にインシデント対応手順が記載されているか
- 依存ライブラリの自動更新アラートが設定されているか

## 9. コード品質

「動く」だけでなく「維持できる」状態でリリースする。
→ 詳細は `principles/code-quality.md` を参照。

```
構造的品質：
  [ ] 200行を超えているファイルがないか（あれば分割を検討）
  [ ] 同じロジックが複数箇所にコピーされていないか
  [ ] マジックナンバーが定数化されているか
  [ ] 循環依存がないか（ESLint import/no-cycle で確認）

保守性の担保：
  [ ] リンター・フォーマッターがCIに組み込まれているか
  [ ] 技術的負債が TODO コメントで可視化されているか
  [ ] コードを読んで各ファイルの責務が1文で説明できるか
```

## リリース判断の基準

### 必須（これがないとリリースしない）
- セキュリティ：認証・HTTPSは必須
- セキュリティ：機密情報の平文保存・コード直書きは禁止
- エラーハンドリング：ユーザーにスタックトレースを見せない
- バックアップ：本番DBのバックアップ設定
- 法的：外部APIライセンスの確認・プライバシーポリシー（個人情報収集がある場合）

### 商用リリースの追加必須項目

個人・小規模の必須項目に加えて、商用サービスでは以下も必須。

```
ガバナンス：
  [ ] ブランチプロテクションが設定されているか（main への直接 push 禁止）
  [ ] CODEOWNERS が設定され、重要ファイルに承認者が割り当てられているか
  [ ] PR テンプレートが用意されているか

SLA / 可用性：
  [ ] SLA / SLO / SLI が定義され、ARCHITECTURE.md に記録されているか
  [ ] エラーバジェットの計算・管理方法が決まっているか
  [ ] ステージング環境が本番と同じ構成で存在するか

インシデント対応：
  [ ] インシデントの重要度定義（P0〜P3）が決まっているか
  [ ] オンコール体制・連絡先が決まっているか
  [ ] ポストモーテムのテンプレートが準備されているか（docs/postmortem-template.md）

監査ログ：
  [ ] 監査すべき操作が定義されているか（ログイン・権限変更・個人情報アクセス等）
  [ ] 監査ログがアプリケーションログとは別に保存されているか
  [ ] 保存期間が法的要件を満たしているか

変更管理：
  [ ] 大規模変更（DBスキーマ・APIバージョニング）のリリース計画が文書化されているか
  [ ] ロールバック手順がテストされているか（演習実施済み）
  [ ] ユーザーへの事前通知フローが決まっているか（メンテナンス通知等）
```

→ 詳細は `principles/commercial-operations.md` を参照。

### 推奨（可能であれば対応する）
- パフォーマンス：インデックス・ページネーション
- 監視：エラー通知・死活監視
- 多層防御：最小権限・依存ライブラリの更新アラート

### 許容可能な妥協（理由をADRに記録する）
- テストカバレッジが低い（理由：プロトタイプ段階）
- 監視が手動確認のみ（理由：ユーザー数が少ない）
- スケーリング設計がない（理由：当面は小規模のため）

## AIへの活用方法

このチェックリストをAIに渡すことで、コードレビューを依頼できる。

```
以下のチェックリスト（production-readiness.md）に基づいて、
このコードが本番運用に耐えられるか評価してください。
未対応の項目と、対応が必要な理由を指摘してください。

[コードを貼り付け]
```
