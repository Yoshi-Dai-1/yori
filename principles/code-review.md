# コードレビュー（Code Review）

個人開発・AI協働開発における、コードの品質を維持するためのレビュー基準。

---

## コードレビューの目的

コードレビューは「間違いを探す作業」ではなく「設計の意図を守る作業」。

チーム開発では人間同士がレビューするが、個人開発・AI協働開発では：
- **AIが書いたコードを人間がレビューする**
- **人間が書いたコードをAIがレビューする**
- **AIが書いたコードをAIがレビューする（自己レビュー指示）**

の3パターンが発生する。それぞれに異なる注意点がある。

---

## レビューの観点（優先順位順）

### 1. 正確性（Correctness）

動作が仕様通りか。バグがないか。

```
確認すること：
[ ] 機能要件（project-definition.md の Must）を満たしているか
[ ] エッジケース（空配列・null・0・負の数・最大値）が考慮されているか
[ ] エラーが適切にハンドリングされているか
[ ] 非同期処理の順序が正しいか（race conditionがないか）
```

### 2. セキュリティ（Security）

脆弱性がないか。

基本的なセキュリティ観点をここで確認する。
詳細なセキュリティ監査（認証・認可の設計・依存ライブラリスキャン・OWASP Top 10対応確認等）は
`@security-auditor`（監査モード）が担う。実装完了後に自律的に呼び出される。

```
確認すること：
[ ] 外部入力がバリデーションされているか
[ ] 機密情報がログに出力されていないか
[ ] 認証・認可が必要な処理に保護があるか
[ ] SQLインジェクション・XSSの対策がされているか
```

### 3. 設計の整合性（Consistency）

ARCHITECTURE.md のルールに従っているか。

```
確認すること：
[ ] 依存の方向が正しいか（上位層が下位層にのみ依存しているか）
[ ] 命名規則（naming-conventions.md）に従っているか
[ ] ファイルサイズが200〜300行以内か
[ ] SSOTが守られているか（同じ情報が複数箇所にないか）
[ ] マジックナンバーが定数化されているか
```

### 4. 可読性（Readability）

名前と構造から意図が読み取れるか。

```
確認すること：
[ ] 関数・変数名が「何をするか」を表しているか
[ ] コメントが「なぜそうするか」を説明しているか（何をするかではなく）
[ ] 深いネストが早期リターンで整理されているか
[ ] 1関数が1つの責務に収まっているか
```

### 5. テスト（Testability）

テストが書けるか・書かれているか。

```
確認すること：
[ ] ビジネスロジックにユニットテストがあるか
[ ] テストが実装の変更で壊れない設計か（実装ではなく振る舞いをテストしているか）
[ ] テストが「Arrange → Act → Assert」の構造になっているか
```

## AIが書いたコードをレビューするときの注意点

AIは「動くコード」を高速に生成するが、以下のパターンで品質が下がりやすい。

**過剰実装（Over-engineering）**
```
症状：要求していない機能・抽象化・最適化が含まれている
対処：project-definition.md の Won't を確認し、不要な実装を削除する
```

**命名の不一致**
```
症状：既存コードと異なる命名スタイルが混入している
     （例：既存は camelCase なのに snake_case の変数が生まれる）
対処：coding-conventions.md を渡してからレビューを依頼する
```

**依存の方向の違反**
```
症状：shared/ が features/ を import している、など
対処：「ARCHITECTURE.md の層のルールに違反していないか確認して」と指示する
```

**エラーハンドリングの省略**
```
症状：try/catch がない・エラーを握りつぶしている
対処：レビュー時に「エラーハンドリングを確認して」と明示的に指示する
```

**マジックナンバーの埋め込み**
```
症状：定数ファイルに移すべき数値がコード中に直書きされている
対処：「定数はすべて constants/ に移して」と指示する
```

## AIにコードレビューを依頼するときの指示テンプレート

```
以下のコードを、dev-standards の規約に基づいてレビューしてください。

確認してほしい観点：
1. 正確性：エッジケースが考慮されているか
2. 設計：ARCHITECTURE.md の層のルールに違反していないか
3. 命名：naming-conventions.md に従っているか
4. SSOT：同じ情報が複数箇所に書かれていないか
5. セキュリティ：外部入力がバリデーションされているか

問題があれば、該当箇所と修正案をセットで指摘してください。
問題がなければ「問題なし」と明示してください。

[コードを貼り付け]
```

---

## 人間が書いたコードをAIにレビューさせるときの追加注意

AIは「問題ない」と言いやすい傾向がある。
以下の指示を加えることで、より厳密なレビューになる：

```
「問題がなさそうに見えても、潜在的なリスクや改善の余地があれば積極的に指摘してください。
 『良さそうです』だけで終わる場合は、具体的に何を確認した上で問題ないと判断したかを説明してください。」
```

## セルフレビューのチェックリスト

実装後・AIへのレビュー依頼前に自分で確認する。

```
[ ] このコードは project-definition.md の Must 要件を満たしているか
[ ] ARCHITECTURE.md の依存ルールに違反していないか
[ ] 命名規則に従っているか
[ ] ファイルサイズが適切か（200〜300行以内）
[ ] マジックナンバーが定数化されているか
[ ] エラーがハンドリングされているか
[ ] 機密情報がハードコードされていないか
```

---

## レビューのタイミング

```
実装前  → project-definition.md・ARCHITECTURE.md をAIと確認する
実装中  → 200行を超えたら分割を検討する（自動的なレビューポイント）
実装後  → セルフレビュー → AIレビュー依頼
リリース前 → production-readiness.md のチェックリストを確認
```
