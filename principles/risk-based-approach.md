# リスクベースアプローチ（Risk-Based Approach）

## このファイルの目的

「強力であればあるほど良い」は間違い。
セキュリティ・コード品質・テスト・ドキュメントすべてに、過剰投資と過少投資がある。

このファイルは「何にどれだけ投資するか」を判断するための思考フレームを提供する。
AIへの指示を書く前に「何が本当に重要か」を決める羅針盤として使う。

---

## リスクベースアプローチとは

```
守るべき資産の価値
────────────────── ＝ 投資すべき水準
攻撃・失敗の確率 × 被害額
```

この式は「感覚で決めない」ために存在する。

例：
- 匿名ユーザーのみの静的情報サイト → セキュリティへの過剰投資は時間の無駄
- 金融資産・医療情報を扱うサービス → セキュリティへの投資不足は致命的

**正しい問いは「どれだけ強力にするか」ではなく「このリスクに見合う投資か」**

---

## リスクの4象限

すべてのリスクをこの枠組みで分類する。

```
            影響が大きい
                 │
    ────────────────────────
    │  C:計画的に対処      │  A:即座に対処（最優先）
    │  影響大・確率低      │  影響大・確率高
    ────────────────────────
    │  D:許容する          │  B:監視する
    │  影響小・確率低      │  影響小・確率高
    ────────────────────────
                 │
            影響が小さい
    確率低                  確率高
```

判断のポイント：
- A領域：今すぐ対処しないとプロジェクトが死ぬ
- B領域：放置すると技術的負債として蓄積する
- C領域：計画に組み込む（次スプリントで対処）
- D領域：記録だけして放置でよい

## 深刻度ラベルの体系と使い分け

このリポジトリでは用途に応じて異なるラベル体系を使用する。
以下の表がSSOTとなり、他ファイルはこの表を参照する。

| ラベル体系 | 用途 | 参照元 |
|-----------|------|--------|
| CRITICAL/HIGH/MEDIUM/LOW | セキュリティ脆弱性・脅威モデリング | rules/security.md, principles/threat-modeling.md |
| A/B/C/D（4象限） | リスク優先順位付け（影響度×確率） | このファイル |
| GREEN/YELLOW/RED | システムレジリエンス診断 | agents/subagents/resilience-checker.md |
| GOOD/CAUTION/ATTENTION | コード品質診断 | agents/subagents/code-quality-auditor.md |

### 4象限とCRITICAL/HIGH/MEDIUM/LOWの紐付け

```
A領域（影響大・確率高）→ CRITICAL（即時対処）
B領域（影響小・確率高）→ HIGH（このスプリント内）
C領域（影響大・確率低）→ MEDIUM（次のスプリントで対処）
D領域（影響小・確率低）→ LOW（記録して計画的に対処）
```

## プロジェクト種別ごとのリスクの重み付け

規模・目的・対象によってリスクの重み付けは変わる。

| プロジェクト種別 | 最重要リスク | 相対的に低いリスク |
|----------------|------------|-----------------|
| 個人開発ツール | 技術的負債・属人化（ドキュメント不足） | コンプライアンス・スケーラビリティ |
| BtoCサービス | セキュリティ・スケーラビリティ・法的リスク | 属人化（個人運営の場合は低い） |
| 社内ツール | 可観測性・依存関係・属人化 | 法的リスク・スケーラビリティ |
| OSS | 知識の集中・依存関係・ライセンス | スケーラビリティ（コミュニティが補う） |
| 医療・金融系 | コンプライアンス・データ損失・セキュリティ | スケーラビリティ（初期） |

## セキュリティへのリスクベースアプローチ

### 「強力であればあるほど良い」が間違いである理由

| セキュリティ強化 | 生まれる代償 |
|---------------|------------|
| MFA必須化 | ユーザーの離脱率上昇 |
| セッション有効期限を極端に短く | 頻繁な再ログインでUX悪化 |
| 過剰な入力制限 | 正当なユーザーが弾かれる |
| 全通信を多重暗号化 | レイテンシ増大・コスト増 |
| 厳格すぎるCSP | 正規サードパーティSDKが動かない |

### 正しい判断基準

```
ユーザーに見えるところ（認証フロー・操作感）
  → 摩擦を最小化する（過剰なセキュリティはUXを壊す）

ユーザーに見えないところ（通信暗号化・DB保護）
  → 妥協しない（ユーザーが気づかないところで守る）
```

### フェーズと規模による優先度

| 規模 | 最優先でやること |
|------|--------------|
| 個人・小規模 | TLS・パスワードハッシュ化・`.env`管理・入力バリデーション |
| 中規模チーム | ↑ + RBAC・JWT適切運用・CIへのセキュリティスキャン統合 |
| 大規模・商用 | ↑ + 脅威モデリング・mTLS・Vault・ペネトレーションテスト |

## コード品質へのリスクベースアプローチ

品質投資にもトレードオフがある。すべてを完璧にしようとすると開発が止まる。

### 品質投資の優先度フレーム

```
投資を惜しまない：
  ビジネスロジックの核心部分のテスト
  認証・決済・データ保護に関わるコードのレビュー
  本番環境に影響するコードの変更

投資を絞る：
  プロトタイプ段階のUIコンポーネント
  一時的なスクリプト・移行ツール
  探索的実装（仕様が固まっていないもの）

記録して放置：
  「いつかリファクタリングしたい」部分
  影響範囲が限定的な技術的負債
```

### テストへのリスクベースアプローチ

「カバレッジ100%」は目標ではない。

```
厚くテストすべき（リスクが高い）：
  - お金・認証・個人情報に関わる処理
  - 複雑なビジネスロジック（条件分岐が多い）
  - 過去にバグが出た箇所

薄くていい（リスクが低い）：
  - 単純なUIコンポーネント（見た目の確認はE2Eに任せる）
  - 外部APIのラッパー（モックとの乖離が起きやすい）
  - 一時的なスクリプト
```

## 「想定できない脅威」への対処

未知の脅威は定義上、チェックリストで防げない。
正しい対処は「防ぎ切ろうとしない」設計。

### 多層防御（Defense in Depth）

4層モデルの定義と各層のチェックリストは resilience.md の「4つの防御層」を参照。
以下の概要は理解のための簡易版。詳細は resilience.md にある。

```
Layer 1: 壊れにくくする（Prevention）    → resilience.md Layer 1 参照
Layer 2: 気づく（Detection）            → resilience.md Layer 2 参照
Layer 3: 被害を限定する（Containment）  → resilience.md Layer 3 参照
Layer 4: 回復する（Recovery）           → resilience.md Layer 4 参照
```

どれか1層が破られても他の層が機能する設計。
**完全な防御は存在しない** という前提に立つことが安全な設計の出発点。

### 未知の脅威への最大の対抗手段

皮肉にも技術的なものではなく **依存ライブラリの定期更新**。
新しく発見された脆弱性のほとんどは、パッチが存在するにもかかわらず
古いバージョンを使い続けていることが原因。

```bash
# 月次で実行する
npm audit          # Node.js
pip-audit          # Python

# CIに組み込む（自動化が理想）
# → 脆弱性が発見されたら自動でアラートが来る
```

## プロジェクト永続性への脅威マップ

脅威はセキュリティだけではない。重み付けして対処する。

```
                    即死系                    緩慢死系
              ┌─────────────┐          ┌─────────────────┐
技術的脅威    │ セキュリティ │          │ 技術的負債       │
              │ 侵害         │          │ スケーラビリティ │
              │ データ消失   │          │ 可観測性欠如     │
              └─────────────┘          └─────────────────┘
              ┌─────────────┐          ┌─────────────────┐
外部的脅威    │ 法的訴訟     │          │ 依存関係リスク   │
              │ 規制違反     │          │ ベンダー変更     │
              └─────────────┘          └─────────────────┘
              ┌─────────────┐          ┌─────────────────┐
組織的脅威    │ キーマン依存  │          │ 属人化           │
              │（知識の集中） │          │ ドキュメント不足 │
              └─────────────┘          └─────────────────┘
```

セキュリティは「即死系」だが、**緩慢死系の脅威こそが長期的には多くのプロジェクトを殺す。**

## AIとの協働でリスク判断をする

### プロジェクト定義時

```
以下のプロジェクト概要（project-definition.md）を読んで、
リスクベースアプローチの観点から以下を評価してください：

1. 最も影響が大きいリスク（即死系）を3つ挙げてください
2. 緩慢死系で見落とされやすいリスクを2つ挙げてください
3. 「やりすぎ」になりやすい投資領域はどこか教えてください

評価後、ARCHITECTURE.md の非機能要件セクションに
反映すべき項目を優先度付きで提案してください。
```

### 実装判断時

```
この機能（[機能名]）を実装するにあたり、
リスクベースアプローチで判断してください：

- この機能が失敗したときの最大の影響は何か
- どの部分のテストを厚くすべきか
- セキュリティ上の懸念はあるか
- 今実装すべきか・後回しにできるか

判断理由とともに教えてください。
```

### 月次の定期診断

```
現在のプロジェクト状態（ARCHITECTURE.md・decisions/・usage/参照）を踏まえて、
リスクの4象限で現状を評価してください：

- A領域（即座に対処）：___
- B領域（監視する）：___
- C領域（計画的に対処）：___
- D領域（許容する）：___

最も対処が急ぎな項目を1つ選び、具体的なアクションを提案してください。
```
