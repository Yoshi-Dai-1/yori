# プロジェクト定義（Project Definition）

開発を始める前に決めること。
技術選定・アーキテクチャ・ディレクトリ構成はすべてここから導かれる。

---

## なぜ先に定義するのか

「何を作るか」が曖昧なまま実装を始めると：
- 技術選定が後から覆る（SSRが不要だったのにNext.jsを選んでいた、など）
- 設計の前提が崩れ、大規模な作り直しが発生する
- AIへの指示が毎回ブレる（文脈が定まっていないため）

AIと開発するとき、この定義ファイルを渡すことで「なぜこのプロジェクトが存在するか」を毎回説明せずに済む。

---

## 定義すべき5項目

### 1. 目的（Why）

このプロジェクトが存在する理由。解決する問題。
「〜できるようにするため」という形で書く。

```
悪い例：「株式分析ツールを作る」（何をするかしか書いていない）
良い例：「ファンダメンタル分析に基づく投資判断を、複数の情報源を行き来せずに
         1画面で完結させるため」（なぜ作るかが明確）
```

### 2. 対象ユーザー（Who）

誰が使うか。自分だけか・チームか・一般公開か。
ユーザーの技術レベル・使用環境（PC/スマホ）・利用頻度も書く。

これによって以下が決まる：
- UIの複雑さ（自分だけなら粗くていい・公開するなら丁寧に）
- 認証の必要性（自分だけなら不要なことが多い）
- パフォーマンス要件（1人なら低スペックでOK）

### 3. 機能要件（What）

何ができるか。MoSCoW法で優先度を整理する。

```
Must（必須）：これがなければ成立しない
  - 株式の基本情報を表示する
  - スクリーニング条件を設定して銘柄を絞り込む

Should（重要だが必須ではない）：
  - スクリーニング条件を保存する

Could（あれば良い）：
  - 銘柄同士を比較する

Won't（今回はやらない）：
  - リアルタイムの株価更新
```

「Won't」を明示することがAI協働で特に重要。
未定義だとAIが過剰実装する（キャッシュ・リアルタイム更新・認証を勝手に追加するなど）。

### 4. 技術制約（Constraint）

選択の自由度を制限する固定条件。

```
言語の制約：
  - すでに習熟している言語がある場合
  - チームの技術スタックに合わせる必要がある場合

環境の制約：
  - デプロイ先が決まっている（GitHub Pages・Vercel・AWS等）
  - 使用できる外部APIが限られている

コストの制約：
  - 月額コストの上限
  - 無料枠内で収める

ライセンスの制約：
  - 商用利用するか否か（MITライセンスのみ使えるか等）
```

### 5. 成功基準（Definition of Done）

「完成」をどう定義するか。数値で表せるものは数値で書く。

```
悪い例：「使いやすいツールが完成したら」
良い例：
  - Must要件がすべて動作する
  - 主要操作が3クリック以内で完了する
  - 自分が毎日実際に使っている状態になっている
```

## AIへの活用方法

このファイルをセッション冒頭に渡すことで、AIが「なぜこれを作っているか」を理解した上でコードを生成する。

特に **Won't** セクションが重要。
「リアルタイム更新は今回やらない」と明示すれば、AIはWebSocketやポーリングの実装を提案しなくなる。

また**成功基準**を渡すことで、「これはMust要件を満たすか」という観点でAIにレビューを依頼できる。

## AIと対話してプロジェクト定義を作る

### なぜ対話で作るのか

テンプレートを人間が独力で埋めると、以下が起きやすい：
- 自分が考えていないことに気づかない（盲点が残る）
- Won'tを書かない（やらないことを明示するのは難しい）
- 「良い例」と同じ粒度で書けているか自分では判断できない

AIにインタビュアーを依頼することで：
- 「なぜ？」「誰が？」「やらないことは？」を引き出される
- 曖昧な記述をその場で指摘・修正できる
- 完成後すぐにAIへ渡せるファイルになる

### 対話プロンプト（新規プロジェクト開始時にそのまま使う）

```
以下の手順でプロジェクト定義を一緒に作ってください。

【手順】
1. 私に質問を1つずつしてください（複数まとめて聞かない）
2. 私の回答をもとに、下記5項目のテンプレートを少しずつ埋めてください
3. 各項目が埋まったら「この内容で合っているか」を確認してください
4. 5項目がすべて埋まったら、完成したテンプレートをMarkdown形式で出力し、`docs/project-definition.md` として保存してください
5. 保存後、`.opencode/standards/architectures/_how-to-choose.md` の選択フローチャートを読み、`docs/project-definition.md` の内容に基づいてプロジェクトに合ったアーキテクチャを自律的に判断し、選定結果と理由を人間に提示してください。人間が承認したら該当のアーキテクチャファイルを読み、`ARCHITECTURE.md` を記入してください

【引き出してほしい項目】
- Why：このプロジェクトが解決する問題・存在する理由
- Who：誰が・どんな環境で・どのくらいの頻度で使うか
- What（Must）：これがなければ成立しない機能
- What（Won't）：今回は絶対にやらないこと（明示が特に重要）
- Constraint：技術・コスト・ライセンスの固定条件
- Security：守るべき資産・認証の必要性・法的制約（1問で引き出す）
- Risk：即死系リスクと許容するリスク（1問で引き出す）
- Definition of Done：「完成」をどう判断するか

【守ってほしいルール】
- Won'tは必ず1つ以上引き出してください
- Security と Risk は必ず引き出してください（「小さいプロジェクトだから不要」とみなさない）
- 「使いやすい」「高速な」などの曖昧な表現があれば指摘して、数値や具体的な状態に直してください
- 成功基準は「自分が毎日使っている状態」のような検証可能な形にしてください
- 質問は1つずつ。私が答えるまで次の質問をしないでください

【出力テンプレート】
本ファイルの「AIと対話してプロジェクト定義を作る」セクションのテンプレートを使ってください。
```

### 完成後の使い方

完成したテンプレートを `docs/project-definition.md` として保存する。

**次のステップ（アーキテクチャ選定）：** `.opencode/standards/architectures/_how-to-choose.md` の選択フローチャートを使ってプロジェクトに合ったアーキテクチャを選び、`ARCHITECTURE.md` を記入する。

以降のセッション開始時に以下の一文を添えるだけで、AIが文脈を理解した状態で作業を開始できる。

```
docs/project-definition.md を読んでから作業を始めてください。
```
