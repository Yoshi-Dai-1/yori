# コード品質（Code Quality）

## このファイルの目的

「動くコード」と「良いコード」は別物。
コード品質の劣化は静かに進行し、気づいたときには手遅れになっている。

このファイルは「何が品質か」を定義し、AIとの協働でその基準を守るための手引き。

---

## コード品質の6つの軸

品質は一言では定義できない。複数の軸が同時に存在する。

### 軸1：可読性（Readability）

「他者（または未来の自分）がコードを読んで意図を理解できるか」

コードは書く時間より読む時間のほうが圧倒的に長い。
読むたびに解読が必要なコードは、変更のたびにコストを生む。

```typescript
// 低品質：何をしているかわからない
const f = (x: number, y: number, z: number) => x * (1 - y) ** z;

// 高品質：意図が自明
const calculateDepreciatedValue = (
  initialValue: number,
  annualRate: number,
  years: number,
): number => initialValue * (1 - annualRate) ** years;
```

判断基準：
- 関数名・変数名だけで「何をするものか」が分かるか
- コメントが「なぜそうするか」を説明しているか（「何をするか」はコードを読めば分かる）
- 深いネストが早期リターンで整理されているか

### 軸2：保守性（Maintainability）

「変更・拡張・バグ修正が安全に行えるか」

低保守性のコードは「触るのが怖い」状態になる。
開発者が変更を避けるようになると、機能追加もバグ修正も停滞する。

低保守性のサイン：
- 1つのファイルがfile-size-and-cohesion.md の閾値を超えている（単一責任の原則違反のサイン）
- 同じロジックが複数箇所にコピーされている（DRY原則違反）
- マジックナンバーが散在している（意味不明な `0.08` や `1440` など）
- 変数名が `a`、`b`、`tmp`、`data2` など

### 軸3：テスト可能性（Testability）

「自動テストを書けるか」

テストが書けないコードには構造的な問題がある。
外部依存が直接埋め込まれていると、テストで差し替えられない。

```typescript
// テストが書けないコード（外部依存が直接埋め込まれている）
function getUserOrders() {
  const db = new Database('production-url'); // 直接DB接続
  const user = getCurrentUser();             // グローバル状態参照
  return db.query(`SELECT * FROM orders WHERE user_id = ${user.id}`);
}

// テスト可能なコード（依存を外から注入できる）
function getUserOrders(db: Database, userId: string) {
  return db.query('SELECT * FROM orders WHERE user_id = ?', [userId]);
}
```

### 軸4：複雑性（Complexity）

「コードの認知的負荷はどの程度か」

循環的複雑度（Cyclomatic Complexity）：分岐の数で測る定量指標。

```
if が 1つ  → 分岐が 2通り
if が 5つ  → 分岐が最大 32通り → テストケースも 32通り必要
if が 10個 → 事実上テスト不可能

目安：1関数の複雑度は 10 以下に保つ
```

複雑度が高くなってきたサイン：
- 関数の責務を20文字以内で説明できない場合（例: `calculateDepreciatedValue` → 「減価償却後の資産価値を計算」はOK、`processData` → 「データを処理する」はNG・抽象的すぎる）
- テストケースを書こうとして途方に暮れる
- バグ修正が別のバグを生む

### 軸5：一貫性（Consistency）

「コードベース全体でスタイル・規約が統一されているか」

フォーマット・命名規則・ファイル構成がバラバラなコードは、
読むたびに脳が再適応するコストを払う。

一貫性はリンター・フォーマッターで機械的に担保する（人間が守ろうとしない）。
→ `coding-conventions.md` と言語固有のリンター・フォーマッター設定が一貫性の実装（JS/TS: ESLint + Prettier、Python: Ruff、Rust: clippy + rustfmt 等）。

### 軸6：依存関係の健全性（Dependency Health）

「モジュール間の依存が適切に管理されているか」

```
悪い依存関係：
UI層 ←→ DB層（相互参照）
問題：どこか1つ変えると全体が壊れる

良い依存関係：
UI層 → ビジネスロジック層 → データアクセス層
変更が局所化される
```

循環依存は設計崩壊のサイン。言語固有のリンターで自動検出する（JS/TS: ESLintの `import/no-cycle`、Python: Ruff の `TCH` / pylint、Go: `go vet` 等）。

## 品質劣化の進行モデル

```
初期  → 「とりあえず動く」コードが積み上がる
中期  → 新機能追加に予想外の時間がかかり始める
後期  → バグ修正が別のバグを生む。誰も全体を把握できない
末期  → 「リファクタリング vs 作り直し」の議論が始まる
       （どちらも完遂できずプロジェクトが止まる）
```

**コード品質の劣化はプロジェクトの緩慢死であり、最も過小評価されているリスク。**

## 設計原則（AIへの指示に使う用語）

### SOLID から特に重要な2つ

**単一責任の原則（SRP）**
「1つの関数・クラスは1つのことだけをする」
→ 関数が長くなってきたら分割のサイン
→ AIへの指示：「この関数を単一責任になるように分割して」

**依存性逆転の原則（DIP）**
「具体的な実装ではなく抽象（インターフェース）に依存する」
→ テスト可能性と保守性が劇的に向上する
→ AIへの指示：「外部依存をインターフェース経由にして、テスト可能な構造にして」

### DRY原則（Don't Repeat Yourself）

「同じロジックを2箇所以上に書かない」
→ ただし、過剰なDRYは逆に複雑性を上げることがある
→ 「2回同じことを書いたら共通化を検討する。3回書いたら必ず共通化する」

### ボーイスカウトルール

「キャンプ場は来た時よりきれいにして去る」
→ 触ったファイルは、触る前より少しだけきれいにして去る
→ 大規模リファクタリングは永遠に来ない。小さな改善の継続が唯一機能する戦略。

### YAGNI（You Aren't Gonna Need It）

「今必要でないものは作らない」
→ AIは過剰実装する傾向がある（キャッシュ・最適化・汎用化を勝手に追加する）
→ AGENTS.md の Boundaries に「指示されていない機能を追加しない」を必ず書く

## ベストプラクティス

### 機械で担保できることは機械に任せる

```bash
# JavaScript/TypeScript
ESLint（ルール違反検出）+ Prettier（フォーマット統一）

# Python
Ruff（高速・フォーマット+lint統合）

# pre-commitフックに仕込む
# コミット時に自動でチェックが走る → 人間が忘れても機能する
```

人間がレビューで指摘するべきことを機械に任せる。
人間は設計・ロジック・意図の確認に集中する。

### テスト戦略：テストピラミッドに従う

```
        /\
       /E2E\         少数（重要なユーザーフローのみ）
      /------\
     /統合テスト\     中程度（APIの境界・DB連携など）
    /------------\
   /  単体テスト  \   多数（ビジネスロジックの核心部分）
  /________________\
```

「テストカバレッジ100%」より「ビジネスロジックの核心部分を厚くテストする」方が
費用対効果が高い。

### 技術的負債を見える化する

```typescript
// 技術的負債は TODO コメントで明示的に記録する
// TODO(dev): このロジックは stock.service.ts の calculate_score() と重複している。
//   統合を検討。 Issue: #142
function calculateLegacyScore(value: number): number {
  ...
}
```

「見えない負債」が「見える負債」になるだけで、対処できるようになる。

## AIとの協働での品質維持

### AIが書いたコードで品質が下がりやすいパターン

```
過剰実装        → 要求していない最適化・抽象化が入る
                  対処：YAGNI原則をBoundariesに明記する

命名の不一致    → 既存コードと異なる命名スタイルが混入する
                  対処：coding-conventions.md を渡してから依頼する

エラーハンドリング省略 → try/catchがない・エラーを握りつぶす
                  対処：「エラーハンドリングを確認して」と明示する

マジックナンバー → 定数ファイルに移すべき数値が直書きされる
                  対処：「定数はすべて constants/ に移して」と指示する

複雑性の蓄積    → 動けばいいコードが積み上がる
                  対処：file-size-and-cohesion.md の閾値に従い分割を指示する
```

### 品質チェックの確認順序（毎実装後）

実装完了後の確認順序（型チェック → lint → テスト → @code-reviewer → 人間レビュー）は
`principles/tdd-with-ai.md` の「TDDと他のツールの組み合わせ」セクションに定義している。
このファイルでは重複して定義しない。

### 品質の劣化を察知する問い

月次または「何か変だ」と感じたときにAIへ：

```
現在のコードベースを読んで、以下の品質問題が発生していないか診断してください：

1. 300行を超えているファイルがあるか（file-size-and-cohesion.md の閾値。ファイル名と行数を列挙）
2. 同じロジックが複数箇所に重複していないか
3. 関数名から責務が読み取れないものがあるか
4. マジックナンバーが定数化されずに残っているものがあるか
5. テストが書かれていないビジネスロジックがあるか

問題があれば、該当箇所と優先度（HIGH/MEDIUM/LOW）をセットで報告してください。
```

## AIへの指示テンプレート（品質を維持しながら実装する）

```
以下の機能を実装してください。

要件：[機能の説明]

品質要件（必ず守ること）：
- 1ファイルは1つの責務のみ（file-size-and-cohesion.md の閾値に従う）
- マジックナンバーは constants/ に定義する
- エラーハンドリングを省略しない
- 既存のコードスタイル・命名規則に合わせる（coding-conventions.md参照）
- 実装前に「この実装で依存の方向に問題はないか」を確認する

実装後に報告すること：
- 変更ファイルと各ファイルの責務
- テスト結果
- 品質上の懸念点（あれば）
```
