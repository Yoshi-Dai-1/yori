## 用語の定義（このファイル内で使用する）

```
「外部通信」
  → アプリケーションのプロセス外への通信すべてを指す
  → 含まれるもの：外部APIサービス・DB（同一サーバーか否かに関わらず）・
    キャッシュサーバー・内部マイクロサービス・メッセージキュー・外部ストレージ

「冪等でない操作」
  → 同じリクエストを2回実行すると、1回実行した場合と異なる結果になる操作
  → 例：決済の実行・注文の確定・メール送信・SMS送信・プッシュ通知送信

「冪等な操作」
  → 同じリクエストを何回実行しても、1回実行した場合と同じ結果になる操作
  → 例：データの取得（GET）・ステータスの確認・冪等キー付きの書き込み
```

## プロジェクト性質による採否の自律判断

AIは以下の判断フローを自律的に実行する。人間が指示する必要はない。

### Step 1：ARCHITECTURE.md から通信の特性を読み取る

```
確認する項目：
  外部API・外部サービスへの通信があるか
    → ARCHITECTURE.md の「外部依存」または「外部サービス」セクションを確認する
    → 記録がない場合は project-definition.md の「機能要件」から判断する

  DBへの通信があるか
    → ARCHITECTURE.md の「データストア」セクションを確認する

  リクエスト頻度の目安はあるか
    → ARCHITECTURE.md の「非機能要件 > パフォーマンス」セクションを確認する

  冪等でない操作が含まれるか
    → project-definition.md の「機能要件 > Must」から判断する
    → 「決済」「注文」「メール送信」「SMS送信」「通知送信」を含む場合は該当する
```

### Step 2：採否を決定する（曖昧な表現を使わない）

```
タイムアウト設定：
  外部通信が1件でもあるプロジェクトに実装する
  外部通信がないプロジェクトにはスキップする

リトライ（指数バックオフ）：
  以下の両方を満たす場合に実装する
    条件1：外部APIサービスへの通信がある（DBのみの場合は別途判断、下記参照）
    条件2：その通信が失敗してもユーザーに即時エラーを返さなくてよい
  DBへのリトライ：一時的な接続失敗（タイムアウト）に限り実装する。
                  クエリエラー・制約違反にはリトライしない。

冪等性の確保：
  以下のいずれかを満たす場合に実装する
    ・決済・注文・課金に関わる書き込み操作がある
    ・メール・SMS・プッシュ通知の送信がある
    ・外部サービスへの書き込みをリトライする設計になっている
  上記のいずれも該当しない場合はスキップする

サーキットブレーカー：
  以下の両方を満たす場合に実装する
    条件1：異なる外部APIサービスへの通信が2種類以上ある
    条件2：1つの外部サービスの障害が他の機能に影響する設計になっている
  条件1・2のいずれかを満たさない場合はスキップする

コネクションプーリング：
  以下のいずれかを満たす場合に実装する
    ・ARCHITECTURE.md の非機能要件で同時接続数が10以上と定義されている
    ・1秒あたり10リクエスト以上のDB・外部API通信が想定されている
  上記のいずれも該当しない場合はスキップする
```

### Step 3：判断結果を ARCHITECTURE.md に記録する

採否と理由を ARCHITECTURE.md の「通信設計」セクションに記録する。
スキップしたものも「スキップ・理由：〇〇」として記録する（意図的な決定として残す）。

```markdown
## 通信設計

| 対策 | 採否 | 設定値・方針 | 採否の根拠 |

| タイムアウト | 採用 | 接続：[秒] 読み取り：[秒] | [根拠] |
| リトライ | 採用/スキップ | [設定値またはスキップ理由] | [根拠] |
| 冪等性 | 採用/スキップ | [方式またはスキップ理由] | [根拠] |
| サーキットブレーカー | 採用/スキップ | [設定値またはスキップ理由] | [根拠] |
| コネクションプーリング | 採用/スキップ | [設定値またはスキップ理由] | [根拠] |

## 共通実装原則（言語・フレームワークを問わず変わらない）

具体的なコードは「使用言語 + 使用フレームワーク + 上記の検索クエリ」で
取得した公式情報に従う。

### タイムアウト設定

```
原則1：接続タイムアウトと読み取りタイムアウトを必ず両方設定する
  接続タイムアウト：サーバーへの接続が確立するまでの上限時間
  読み取りタイムアウト：接続確立後にレスポンスを受け取るまでの上限時間
  この2つは別の障害を想定している。片方だけの設定は不完全。

原則2：タイムアウト値は constants/ に定数として定義する
  コードに数値を直接書かない（ssot-and-constants.md の原則に従う）
  定数名の例：
    EXTERNAL_API_CONNECT_TIMEOUT_MS
    EXTERNAL_API_READ_TIMEOUT_MS
    DB_CONNECT_TIMEOUT_MS

原則3：タイムアウト値の初期値は検索した公式情報で確認する
  AIの学習データの固定値をそのまま使わない
  「[使用フレームワーク] http client timeout recommended values [現在年]」で検索する
  検索結果を📋フォーマットで人間に通知してから定数値を決定する
```

### リトライ（指数バックオフ）

```
原則1：指数バックオフとジッターを組み合わせる
  固定間隔のリトライは使わない
  理由：複数クライアントが同時に失敗した場合、
        固定間隔だと全員が同時にリトライしてサーバーが過負荷になる
        ジッターを加えることでリトライのタイミングを分散させる

原則2：リトライする条件とリトライしない条件を明示的に分ける
  リトライする条件（一時的な障害）：
    ・ネットワークタイムアウト
    ・HTTP 503（サービス一時停止）
    ・HTTP 429（レートリミット超過）→ Retry-After ヘッダーがある場合はその値を待つ
  リトライしない条件（永続的なエラー・冪等でない操作）：
    ・HTTP 400（リクエスト不正）
    ・HTTP 401（認証失敗）
    ・HTTP 403（認可失敗）
    ・HTTP 404（リソース未存在）
    ・冪等でない操作（決済実行・メール送信など）→ 冪等性の確保で別途対応する

原則3：最大リトライ回数と待機時間の上限を必ず設定する
  上限なしのリトライは設計しない
  設定値は定数として constants/ に定義する
  定数名の例：
    EXTERNAL_API_MAX_RETRY_COUNT
    EXTERNAL_API_RETRY_INITIAL_WAIT_MS
    EXTERNAL_API_RETRY_MAX_WAIT_MS
```

### 冪等性の確保

```
原則1：冪等でない操作のリトライには必ず冪等キーを使う
  冪等キーとは：クライアントが生成する一意のID。
               サーバーが同じIDの2回目以降のリクエストを
               最初の処理結果を返すことで処理を1回に限定する仕組み。

原則2：冪等キーの実装方式を明示する
  方式A：リクエストヘッダーに付与する
    例：Idempotency-Key: [UUID]
    サーバーはこのIDをキャッシュして2回目以降は最初の結果を返す
  方式B：DBのユニーク制約を使う
    同じデータを2回INSERTしても1件になるようにDB制約を設ける
  使用する方式は実装前に決定して ARCHITECTURE.md に記録する

原則3：決済サービスの冪等性仕様は必ず公式ドキュメントで確認する
  各決済サービスによって冪等キーの実装方法が異なる
  「[決済サービス名] idempotency [現在年]」で必ず検索する
  検索結果を📋フォーマットで人間に通知してから実装する
```

### サーキットブレーカー

```
原則1：3つの状態を必ず実装する
  Closed（閉）：正常状態。リクエストを通す。失敗回数を計測する。
  Open（開）：遮断状態。リクエストを通さずに即時エラーを返す。
  Half-Open（半開）：回復確認状態。試験的にリクエストを1件通す。

原則2：状態遷移の閾値を定数として定義する
  定数名の例：
    CIRCUIT_BREAKER_FAILURE_THRESHOLD（Closed→Open の失敗回数）
    CIRCUIT_BREAKER_RESET_TIMEOUT_MS（Open→Half-Open の待機時間）
  設定値は「[使用言語] circuit breaker library [現在年]」で検索した公式情報に従う

原則3：Open状態のときのフォールバック動作を定義する
  フォールバックなし：即時エラーを返す（最もシンプル）
  フォールバックあり：キャッシュ値を返す・デフォルト値を返す
  どちらにするかは実装前に決定して ARCHITECTURE.md に記録する
```

### コネクションプーリング

```
原則1：プール数の設定値は公式ドキュメントで確認する
  「[使用DB名] connection pool recommended size [現在年]」で検索する
  AIの学習データの固定値をそのまま使わない

原則2：プール数を定数として定義する
  定数名の例：
    DB_CONNECTION_POOL_MIN
    DB_CONNECTION_POOL_MAX
    EXTERNAL_API_CONNECTION_POOL_SIZE

原則3：プールの枯渇時の動作を定義する
  タイムアウトまで待機する（タイムアウト値を定数で定義する）
  即時エラーを返す
  どちらにするかは実装前に決定する
```

## WebSocket・リアルタイム通信の設計原則

WebSocketを使用するプロジェクトでのみ適用する（HTTP APIのみのプロジェクトはスキップ）。
変化しない原則のみを定義する。ライブラリ固有の実装はブロックC（フレームワーク固有設計の深掘り）が自律検索して補完する。

```
再接続戦略：
  クライアントは接続が切れたら自動再接続する
  再接続間隔は指数バックオフ（1秒 → 2秒 → 4秒 → 最大30秒）を使う
  無限再接続は避ける（最大試行回数または最大待機時間を設定する）

Heartbeat（KeepAlive）：
  接続が生きているか確認するPing/Pongを定期的に送信する（推奨：30秒間隔）
  Pongが返ってこない場合は接続を切断して再接続する

メッセージの順序保証：
  WebSocketはメッセージの順序を保証しない場合がある
  順序が重要なデータにはシーケンス番号を付与する

スケールアウト時の注意：
  複数サーバーで負荷分散する場合、同じクライアントが毎回同じサーバーに接続する
  設計が必要（スティッキーセッション または Redis Pub/Sub 等のブロードキャスト）
   具体的な実装はブロックC（フレームワーク固有設計の深掘り）が自律検索する：
  `[使用フレームワーク] WebSocket scaling [現在年]`
```

## GraphQL通信の設計原則

GraphQLを使用するプロジェクトでのみ適用する（REST APIのみのプロジェクトはスキップ）。
セキュリティ対策（Introspection制御・クエリ深さ制限・認可・エラーメッセージ）の詳細は
security-implementation.md の「GraphQLセキュリティ」セクションを参照。

このファイルでは通信設計の観点のみを定義する:

```
N+1問題の防止:
  GraphQLはネストしたクエリでN+1クエリが発生しやすい
  DataLoader（またはそれに相当するバッチ処理）を使いDBへのクエリをまとめる
  パフォーマンス問題が発生したら真っ先にN+1を疑う

クエリの深さ制限:
  ネストが深すぎるクエリはサーバーに過負荷をかける
  クエリの深さ・フィールド数・複雑度に上限を設定する

Introspectionの制御:
  Introspection（スキーマ情報の取得）は開発環境のみ有効にする
  本番環境ではIntrospectionを無効化してスキーマ情報を隠す
  → security-implementation.md の「GraphQLセキュリティ」セクションも参照する

Subscriptionのスケール:
  GraphQL Subscriptionを使う場合はWebSocketと同じスケール設計が必要
  → 上記のWebSocket設計原則を参照する
```

## 月次診断での確認（@resilience-checker から参照）

```
月次の定期診断で確認する項目：

1. ARCHITECTURE.md の「通信設計」テーブルが記入されているか
   → 未記入：通信設計が未定義のまま実装されている可能性がある

2. タイムアウト値が constants/ に定数として定義されているか
   → 数値がコードに直書きされている場合は定数化を指示する

3. リトライの対象・非対象が明確に分かれているか
   → 冪等でない操作（決済・メール送信など）がリトライ対象に含まれていないか確認する

4. 冪等でない操作に冪等性の確保が実装されているか
   → project-definition.md の機能要件に決済・通知が含まれる場合は必ず確認する

5. 採用した対策の設定値が constants/ に定義されているか
   → 定数でない設定値が発見された場合は定数化を指示する

報告形式（resilience-checker.md の形式に統合して報告する）：

[OK]     通信設計：ARCHITECTURE.md の通信設計テーブルが記入されている
[要対応] タイムアウト：constants/ に定数が定義されていない → 定数化を指示する
[要対応] 冪等性：決済処理にリトライが実装されているが冪等性の確保がない
[OK]     リトライ条件：冪等でない操作はリトライ対象から除外されている
```
