# Plugin Events: session.deleted の代替案

`handoff.ts` は `session.deleted` イベントでセッション終了を検知する設計だが、
このイベントがファイル書き込み可能なタイミングで発火するか未確認。

## 検証方法

1. `session.deleted` で handoff が生成されるか確認する
2. 生成されない場合、以下を順に試す

## 代替案（試行順）

### 案A: `session.idle`

セッションがアイドル状態になったときに発火する。
通知プラグインの公式サンプルで使用されている実績あり。

**問題点**: メッセージごとに発火するため、重複防止が必要。
`HANDOFF_FILLED` マーカーですでに重複防止は実装済みだが、
セッション途中のアイドルでもマーカーなし状態で更新してしまう。

**対策**: handoff 生成の条件を「編集中のファイルがない」
（= git diff --name-only が空）場合に限定することで、
セッション終了時のみ生成するよう近似できる。

### 案B: `session.updated`

`session.created` / `session.deleted` 以外のすべての更新で発火。
`session.idle` よりは頻度が低い可能性があるが、まだ発火頻度は高い。

**問題点**: 案Aと同じく発火頻度が高い。
**対策**: 発火間隔を制御する仕組み（前回生成からN分以上経過）を併用する。

### 案C: 定期チェック（cron 相当）

`experimental.session.compacting` イベントがコンテクスト圧縮時に発火する。
セッションが長期間続いて圧縮が発生したタイミングで handoff を生成する。

**問題点**: 短いセッションでは圧縮が発生せず handoff が生成されない。
**対策**: 圧縮 + セッション終了の両方で生成する。

## 決定（初版）

初回リリースは `session.deleted` で実装する。
動作確認後、上記代替案へのフォールバックを検討する。

## 決定（2026-07-01 改訂）

`session.deleted` は通常運用でほぼ発火しないデッドコードであることが判明。
以下の設計で全面改訂した：

- **イベント**: `session.idle`（30分デバウンス）
- **AI呼出**: `client.session.prompt({ noReply: true })` で会話履歴に注入
- **Fallback**: `.opencode/.handoff-trigger` ファイル（次セッション開始時に検出）
- **Skill 併用**: 明示的なユーザー終了意図 + タスク完了 + Todo達成時にも発動
- **重複防止**: `<!-- HANDOFF_FILLED -->` マーカーで両者を排他

この改訂により、空テンプレート（機械的情報のみ）は廃止し、AI による complete handoff 生成に統一した。

## 不採用：`todo.updated` Plugin 購読（2026-07-01 検討・却下）

**経緯**: 議論の中で OpenCode 組み込み Todo（AI の `todowrite` ツール）の完了を Plugin で検出できないか検討した。公式イベント一覧に `todo.updated` が存在する。

**却下理由**:
1. **AI が自明に把握**: `todowrite` は AI 自身が呼び出すツール。AI はすべての Todo が完了したことを自明に知っており、SKILL.md の指示で検出可能。
2. **コード購読の必要性なし**: AI が会話履歴から判断できる情報を Plugin で横取り購読するのは冗長。SKILL.md の AI 指示で十分。
3. **イベントプロパティ未確認**: `todo.updated` が完了/作成/更新のどの状態変化まで含むか、完了情報をプロパティとして持つかが未確認。実装前に検証が必要。

**代案としての位置づけ**: 現在の Skill による AI 検出が機能しない（AI が Todo 完了を見逃す）場合のフォールバックとして再検討可能。その場合は `todo.updated` のイベントプロパティ調査が前提。
