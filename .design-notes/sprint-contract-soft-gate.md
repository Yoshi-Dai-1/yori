# Sprint Contract 承認：ソフトゲート設計の判断記録

<!-- 2026-06-04 作成 -->
<!-- 調査結果（features→tasks リネーム後の整合性チェック）で発見・記録 -->

## 背景

2026-06 の整合性調査で「Sprint Contract 承認はプラグインによる機械的強制がない（ソフトゲート）」が設計上の留意点として発見された。

## 現状の設計

```
evaluator.md line 63:
「このステップが完了（承認）するまで Generator は実装を開始しない。」

この指示は AI が evaluator.md を読み、理解し、従うことを前提とする。
```

### ソフトゲートの仕組み

| 要素 | 内容 |
|------|------|
| 判断基準 | evaluator.md の言語指示 |
| 強制手段 | AI への読み込みと理解への依存 |
| ブロック機能 | なし（プラグインによる `tool.execute.before` ガードなし） |
| 検証方法 | docs/spec.md の「承認」欄に `Evaluator / YYYY-MM-DD` が記入されたか |

### ハードゲートとの比較

| 項目 | ソフトゲート（現状） | ハードゲート（仮） |
|------|---------------------|-------------------|
| 仕組み | AI への言語指示のみ | プラグインによる `tool.execute.before` |
| ブロック力 | なし（AI の規則遵守に依存） | あり（機械的にブロック） |
| 複雑性 | 低い | 承認状態の検出・スプリント紐付けが必要 |
| 柔軟性 | 高い（人間の例外指示に対応可能） | 低い（プラグインが全ケースを網羅する必要あり） |
| メンテナンスコスト | 低い | 中程度 |

## あえてハードゲートにしない理由

| # | 理由 | 説明 |
|---|------|------|
| 1 | 複雑性 | 承認状態の検出・スプリントとファイルの紐付けが非自明 |
| 2 | 柔軟性 | 人間が「承認なしで先行して実装して」と指示するケースに対応できない |
| 3 | コスト | プラグインのメンテナンスコストに見合う効果が不明（現状で違反が観測されていない） |
| 4 | 段階的設計 | Phase 1 では最少のハーネスで開始し、問題が観測されてから強化する方針 |

## 残存リスク

| リスク | 発生条件 | 影響 |
|--------|----------|------|
| コンテキスト忘却 | 長時間セッションで evaluator.md の指示を忘却 | 承認前に実装開始 |
| 自己評価バイアス | 「承認がなくても大丈夫」と AI が判断 | 承認プロセスをスキップ |
| Context Anxiety | 手順を省略して完了しようとする | Sprint Contract レビューを飛ばす |

## 監視指標（将来観測時）

- 「承認前に実装を開始した」回数
- 「承認なしで完了宣言した」回数
- `docs/spec.md` の「承認」欄が空のままスプリントが完了した回数

## ハードゲート化の検討トリガー

以下が観測された場合にハードゲート化を検討する：

1. 承認前に実装が開始されるインシデントが発生
2. Sprint Contract レビューなしで PASS 判定が出る
3. Context Anxiety の兆候が頻発する（harness-health.ts で検知）

## ハードゲート化の候補パターン

### パターンA：マーカーファイル方式

```
1. @evaluator が承認 → .opencode/.sprint-approved-N マーカーを作成
2. プラグイン（sprint-gate.ts）が src/** への書き込みを検知
3. 該当スプリントの承認マーカーを確認
4. 未承認ならブロック
```

- evaluator-tools.ts と同パターン（マーカー作成→検出→削除）
- 既存アーキテクチャとの一貫性あり

### パターンB：tasks.json 拡張方式

```
1. tasks.json に sprintApproved: boolean フィールドを追加
2. @evaluator が承認 → sprintApproved を true に更新（tasks-guard.ts と同保護）
3. 実装プラグインが src/** への書き込み時に sprintApproved を確認
```

- tasks.json への一元管理（状態が分散しない）
- ただし既存の tasks-guard.ts と保護ロジックが重複

### 推奨

**パターンA（マーカーファイル方式）** を推奨する。理由：
- 既存の evaluator-tools.ts / tasks-guard.ts のパターンと一貫
- tasks.json の構造変更なし
- プラグインの独立性を維持

## 決定事項

| 判断 | 理由 |
|------|------|
| Phase 1 ではソフトゲートを維持 | 実害なし・複雑性増加を避ける |
| ハードゲート化は問題発生時に検討 | Phase 2 以降の設計判断とする |
| パターンAを候補として記録 | マーカーファイル方式（既存パターンとの一貫性） |
